## 1. 项目概述

### 1.1 项目背景

**项目名称**：电商小二智能客服助手

**业务场景**：面向电商平台用户，提供多轮客服对话能力。用户可以咨询订单状态、查询物流、申请退款、了解商品信息、咨询平台政策，也可以点击前端订单卡片或商品卡片，把对象信息传给后端继续对话。

**核心价值**：

- **多轮任务处理**：支持查订单、查物流、退款申请、商品推荐、转人工等电商客服流程
- **LLM 意图规划**：使用大模型判断用户当前消息属于 task、knowledge 还是 chitchat 轨道
- **YAML 流程编排**：把确定性业务流程写在配置文件里，便于维护和讲解
- **Action 插件机制**：用 action 承接业务接口调用、槽位更新和话术生成
- **任务中断与恢复**：用户可以从退款流程切到物流查询，再恢复之前的退款流程
- **对象消息处理**：前端点击订单/商品卡片后，后端能自动填充订单号或商品 ID
- **历史对话展示**：后端返回带 `session_id` 的历史消息，前端根据页面生命周期插入分割线
- **数字人播报**：前端接入 `lm-avatar-chat-sdk`，后端提供数字人会话和 WebSocket 对话通道，让机器人回复可以驱动数字人开口

### 1.2 技术选型

| 类别       | 技术栈                                  | 选型理由                                                   |
| ---------- | --------------------------------------- | ---------------------------------------------------------- |
| Web 框架   | FastAPI                                 | 异步接口开发简单，依赖注入清晰                             |
| 状态持久化 | MySQL + SQLAlchemy Async                | 持久化用户完整对话状态，支持异步读写                       |
| 对话状态   | 自定义 `DialogueState` 聚合根           | 统一保存任务、槽位、session、turn、聚焦对象                |
| 流程编排   | YAML + 自研 `FlowExecutor`              | 业务流程确定性强，配置化更直观                             |
| LLM 调用   | LangChain PromptTemplate + OutputParser | 用于意图规划、话术改写、知识回复                           |
| 模板渲染   | Jinja2                                  | 用槽位和上下文渲染客服话术                                 |
| HTTP 调用  | httpx 异步客户端                        | 调用电商订单、物流、商品等业务接口                         |
| 前端联动   | Vue + 对象消息                          | 支持订单卡片、商品卡片点击填槽                             |
| 数字人渲染 | 阿里云灵眸 + `lm-avatar-chat-sdk`       | 前端云渲染数字人，后端返回 sessionId 和 RTC 参数           |
| 实时通道   | WebSocket                               | 数字人模式下承载用户文本、机器人文本、打断和 turn_end 事件 |

### 1.3 项目亮点

```text
LLM 只负责规划，不直接执行业务动作
TurnPlanValidator 做结构校验，降低大模型幻觉影响
YAML 定义业务流程，Python Action 执行业务能力
支持 active_task / paused_tasks，实现任务中断和恢复
对象消息和 focused_object 机制支持卡片点击填槽
DialogueState 整体 JSON 持久化，恢复多轮上下文简单
历史接口返回 session_id，前端负责展示分割线
数字人模式复用同一套 DialogueService，只是通过 WebSocket 把 bot_text 推给前端 SDK 播报
```

这套设计的核心思想是：**大模型负责理解，后端代码负责约束和执行**。

## 2. 整体架构设计

### 2.1 请求处理链路

用户每发一条消息，后端都会走一遍完整的对话处理链路。

```text
POST /api/chat
    ↓
chat_router.py
    ↓
_build_user_message：把接口请求转成 UserMessage
    ↓
DialogueService.handle_message
    ↓
DialogueStateRepository.load：加载用户状态
    ↓
DialogueEngine.handle_dialogue：处理当前 turn
    ↓
TurnPlanner：LLM 规划轨道和命令
    ↓
TurnPlanValidator：校验规划结果
    ↓
TaskHandler / KnowledgeHandler / ChitChatHandler
    ↓
DialogueState.commit_turn：提交本轮对话
    ↓
DialogueStateRepository.save：持久化状态
    ↓
ChatResponse 返回前端
```

这个链路里有两个边界很重要：

1. **接口层只做模型转换**，不写业务逻辑。
2. **Engine 只做调度**，具体业务交给 handler、executor、action。

### 2.2 三条处理轨道

电商小二不是所有问题都走同一套逻辑，而是把用户输入分成三类处理。

| 轨道 | 负责场景 | 典型例子 |
| --- | --- | --- |
| Task | 有明确业务流程的任务 | 查订单、查物流、退款、转人工 |
| Knowledge | 信息咨询类问题 | 退款政策、商品信息、配送规则 |
| Chitchat | 闲聊或寒暄 | 你好、谢谢、你是谁 |

对应代码位置：

```text
atguigu/engine/dialogue_engine.py
atguigu/task/handler.py
atguigu/knowledge/handler.py
atguigu/chitchat/handler.py
```

### 2.3 核心状态流转

项目不是无状态问答，而是一个**多轮状态机**。

```text
DialogueState
  ├── active_task          当前正在执行的业务任务
  ├── paused_tasks         被中断的任务栈
  ├── active_system_task   系统提示流程，如收集槽位、任务中断提示
  ├── focused_object       前端当前选中的订单/商品对象
  ├── sessions             用户所有业务会话
  ├── current_session_id   当前业务 session
  └── pending_turn         当前正在处理但还没提交的一轮对话
```

面试时可以这样概括：

> 这个项目把一次客服对话抽象成 DialogueState 聚合根。每个用户一份状态，里面既保存任务执行状态，也保存历史 session 和 turn。每次请求先 load 状态，处理完本轮后再 save 回数据库。

## 3. 核心模块实现

### 3.1 API 与 Service 层

#### 核心职责

1. 接收前端请求
2. 把请求模型转换成领域模型
3. 调用 DialogueService
4. 把领域结果转换成接口响应

核心接口在 `chat_router.py`：

```python
@router.post("/api/chat")
async def chat_endpoint(
        chat_request: ChatRequest,
        service: DialogueService = Depends(get_dialogue_service)
) -> ChatResponse:
    user_message = _build_user_message(chat_request)
    process_result: ProcessResult = await service.handle_message(user_message)
    return _build_chat_response(process_result)
```

`_build_user_message` 会根据请求里是否有 `text` 来判断消息类型：

```python
return UserMessage(
    sender_id=chat_request.sender_id,
    message_id=chat_request.message_id if chat_request.message_id else str(uuid.uuid4()),
    type=MessageType.TEXT if chat_request.text else MessageType.OBJECT,
    text=chat_request.text,
    object=FocusedObject(
        id=chat_request.object.id,
        type=chat_request.object.type,
        title=chat_request.object.title,
        attributes=chat_request.object.attributes,
    ) if chat_request.object else None
)
```

这里支持两类用户消息：

| 类型 | 来源 | 说明 |
| --- | --- | --- |
| TEXT | 输入框文本 | 用户自然语言输入 |
| OBJECT | 前端卡片点击 | 用户点击订单或商品对象 |

Service 层负责加载状态、调用引擎、保存状态：

```python
async def handle_message(self, user_message: UserMessage) -> ProcessResult:
    dialogue_state = await self.dialogue_repository.load(user_message.sender_id)
    process_result: ProcessResult = await self.dialogue_engine.handle_dialogue(dialogue_state, user_message)
    await self.dialogue_repository.save(dialogue_state)
    return process_result
```

这层设计的好处是：接口层很薄，状态读写集中在 service 里，业务处理集中在 engine 里。

### 3.2 DialogueState 状态模型

`DialogueState` 是整个项目最核心的数据结构。

```python
@dataclass(slots=True)
class DialogueState:
    sender_id: str
    active_task: TaskContext | None = None
    paused_tasks: list[TaskContext] = field(default_factory=list)
    active_system_task: SystemContext | None = None
    focused_object: FocusedObject | None = None
    sessions: list[Session] = field(default_factory=list)
    current_session_id: str | None = None
    pending_turn: Turn | None = None
```

字段含义：

| 字段 | 说明 |
| --- | --- |
| `active_task` | 当前正在跑的业务任务，如退款申请 |
| `paused_tasks` | 被打断的任务栈，支持恢复 |
| `active_system_task` | 系统流程，如“请提供订单号” |
| `focused_object` | 前端当前点击的订单或商品 |
| `sessions` | 用户所有历史业务 session |
| `current_session_id` | 当前 session 的 ID |
| `pending_turn` | 当前正在处理的一轮对话，还没提交 |

Session 和 Turn 结构如下：

```python
@dataclass(slots=True)
class Session:
    session_id: str
    started_at: float
    last_activity_at: float
    closed_at: float | None = None
    turns: list[Turn] = field(default_factory=list)

@dataclass(slots=True)
class Turn:
    turn_id: str
    user_message: UserMessage
    bot_messages: list[BotMessage]
```

为什么这样设计？

- `Session` 表示一段连续客服会话
- `Turn` 表示一问一答或一问多答
- `DialogueState` 表示一个用户的完整对话聚合

持久化时，项目没有把 session、turn、slot 拆成多张表，而是整体序列化成 JSON：

```python
state_json: str = json.dumps(dialogue_state.to_dict(), ensure_ascii=False)
```

然后通过 MySQL 主键重复更新实现 upsert：

```python
insert_stmt = insert(DialogueStateRecord).values(
    sender_id=dialogue_state.sender_id,
    state_json=state_json
)

update_stmt = insert_stmt.on_duplicate_key_update(
    state_json=insert_stmt.inserted.state_json
)
```

面试时可以说：

> 这里把 DialogueState 当成聚合根整体持久化，优点是恢复多轮上下文非常简单，active_task、paused_tasks、slots、focused_object、sessions 都能一次恢复。缺点是对单条 turn 的查询分析不如关系型拆表方便。

### 3.3 DialogueEngine 调度中心

`DialogueEngine` 是整个对话的调度中心。

核心流程：

```python
async def handle_dialogue(self, state: DialogueState,
                          user_message: UserMessage) -> ProcessResult:
    self._prepare_session(state)
    self._begin_turn(state, user_message)

    if user_message.type is MessageType.TEXT:
        msgs = await self._handle_text_msg(state, self.turn_planner,
                                           self.task_handler.flows,
                                           self.knowledge_handler.knowledge_intents)
    else:
        state.set_focused_object(user_message.object)
        msgs = await self._handle_obj_msg(user_message, state, self.task_handler.flows)

    state.pending_turn.bot_messages.extend(msgs)
    state.commit_turn()

    return ProcessResult(
        sender_id=user_message.sender_id,
        message_id=user_message.message_id,
        messages=msgs
    )
```

可以拆成六步：

| 步骤 | 说明 |
| --- | --- |
| 1. 准备 session | 没有 session 就新建，超时则关闭旧 session 并新建 |
| 2. 开始 turn | 把用户消息写入 `pending_turn` |
| 3. 判断消息类型 | 文本走 LLM planner，对象走对象解析 |
| 4. 调度 handler | task / knowledge / chitchat / clarify |
| 5. 写入 bot 消息 | 把回复追加到当前 turn |
| 6. 提交 turn | pending_turn 进入当前 session.turns |

#### session 超时处理

```python
if current_session is None:
    state.start_session()
    return

now = time.time()
if now - current_session.last_activity_at > 60 * 60:
    state.close_session()
    state.reset_running_state_for_new_session()
    state.start_session()
else:
    current_session.last_activity_at = now
```

这里用 60 分钟作为业务 session 边界。超时后会：

1. 关闭当前 session
2. 清理正在运行的任务状态
3. 开启新 session

这保证了用户隔很久再回来时，不会继续执行一个很久以前的退款或物流流程。

### 3.4 TurnPlanner 与校验器

文本消息进入 `_handle_text_msg` 后，先交给 `TurnPlanner`。

```python
turn_plan = await turn_planner.predict(state, flows=flows, intents=knowledge_intents)
validated = self.turn_validator.validate(state, turn_plan, flow_list=flows, intents=knowledge_intents)
```

Planner 的职责不是执行任务，而是让 LLM 输出结构化计划。

计划大概分为三种：

```json
{
  "task": null,
  "knowledge": null,
  "chitchat": null
}
```

实际运行时，只能命中其中一个轨道。

Task 轨道会返回命令，例如：

- `start_flow`
- `set_slots`
- `resume_flow`
- `cancel_flow`

Knowledge 轨道会返回意图，例如：

- `product_info`
- `order_info`
- `refund_policy`
- `return_policy`
- `shipping_policy`

为什么还要 `TurnPlanValidator`？

因为 LLM 输出不一定完全可信，必须先校验再执行。

核心校验规则：

```python
active_tracks = self._active_tracks(turn_plan)

if not active_tracks:
    return self._reject(ClarifyReason.MISSING_TRACK)

if len(active_tracks) > 1:
    return self._reject(ClarifyReason.MULTIPLE_TRACKS)
```

Task 轨道校验：

```python
allowed = (StartFlowCommand, SetSlotsCommand, ResumeFlowCommand, CancelFlowCommand)

if not all(isinstance(cmd, allowed) for cmd in task_track.commands):
    return self._reject(ClarifyReason.INVALID_TASK_COMMANDS)

if len(start_flow_cmd) > 1:
    return self._reject(ClarifyReason.MULTIPLE_TASK_FLOWS)

if start_flow_cmd:
    flow_id = start_flow_cmd[0].flow
    flow = flow_list.get_flow_by_id(flow_id)
    if flow is None:
        return self._reject(ClarifyReason.UNKNOWN_TASK_FLOW)
```

Knowledge 轨道校验：

```python
for intent in knowledge_plan.intents:
    intent_meta = intents[intent]
    required_object = intent_meta.requires_object
    if required_object is not None:
        if focused_object is None or focused_object.type != required_object:
            return self._reject(ClarifyReason.MISSING_FOCUSED_OBJECT)
```

面试时可以这样回答：

> 我们没有让 LLM 直接调用业务接口，而是让它输出结构化 TurnPlan。后端再通过 Validator 校验轨道唯一性、命令白名单、流程是否存在、知识意图是否需要聚焦对象。校验失败时走 ClarifyResponder，让系统追问用户，而不是盲目执行。

### 3.5 YAML 流程编排

业务流程定义在：

```text
flow_config/user_flows.yml
flow_config/system_flows.yml
```

`user_flows.yml` 负责用户业务流程，例如：

- `order_status_query`：订单状态查询
- `logistics_tracking`：物流查询
- `refund_request`：退款申请
- `similar_product_recommendation`：相似商品推荐
- `human_handoff`：转人工

以订单状态查询为例：

```yaml
order_status_query:
  name: 订单状态查询
  description: 帮用户查询订单当前的处理状态，例如待付款、待发货、已发货或已完成。
  steps:
    - id: start
      type: start
      next: ask_order_number

    - id: ask_order_number
      type: collect
      slot_name: order_number
      response:
        text: "请告诉我你的订单号。"
      next: lookup_order_status

    - id: lookup_order_status
      type: action
      action: action_lookup_order_status
      next: show_order_status

    - id: show_order_status
      type: action
      action: action_response
      args:
        text: "订单{{ slots.order_number }}当前状态是：{{ slots.order_status }}。{{ slots.order_summary }}"
      next: end

    - id: end
      type: end
      next: [ ]
```

这个流程很典型：

```text
start
  ↓
collect order_number
  ↓
action_lookup_order_status
  ↓
action_response
  ↓
end
```

YAML 中的 step 类型：

| 类型 | 作用 |
| --- | --- |
| `start` | 流程入口，通常只推进到下一步 |
| `collect` | 收集槽位，缺失时触发系统收集流程 |
| `action` | 调用具体 action 执行业务逻辑或回复 |
| `end` | 结束当前系统流程或业务流程 |

`system_flows.yml` 负责系统提示流程，例如：

- `system_task_started`：开始新任务提示
- `system_task_resumed`：恢复任务提示
- `system_collect_information`：收集槽位提示
- `system_task_interrupted`：任务被中断提示
- `system_task_canceled`：任务取消提示
- `system_cannot_handle`：无法处理提示

例如收集信息流程：

```yaml
system_collect_information:
  description: Flow for asking the user for a slot value during a collect step
  name: collect information
  steps:
    - id: start
      type: start
      next: ask

    - id: ask
      type: action
      action: action_response
      args: context.response
      next: listen

    - id: listen
      type: action
      action: action_listen
      next: end

    - id: end
      type: end
      next: []
```

这里的 `action_listen` 是一个哨兵动作，表示系统说完了，该等用户输入了。

面试时可以强调：

> 用户业务流程和系统提示流程分开配置。业务流程描述“要完成什么任务”，系统流程描述“任务切换、收集信息、恢复任务时怎么说”。这样避免把过场话术写死在 Python 代码里。

### 3.6 CommandProcessor 命令处理

LLM planner 输出的是命令，不是直接修改状态。

命令处理由 `CommandProcessor` 完成：

```python
def run(self,
        state: DialogueState,
        commands: List[Command],
        flow_list: FlowsList
        ) -> None:
    for command in commands:
        self._apply(state, command=command, flow_list=flow_list)
```

支持四种命令：

| 命令 | 作用 |
| --- | --- |
| `StartFlowCommand` | 开启一个业务流程 |
| `SetSlotsCommand` | 给当前任务填槽 |
| `ResumeFlowCommand` | 恢复暂停任务 |
| `CancelFlowCommand` | 取消当前任务 |

```python
self._activate_start_system_task(
    state,
    flow_list,
    started_flow_id=command.flow,
    started_flow_name=self._readable_flow_name(command.flow, flow_list)

```

#### 3.6.2 中断任务

如果用户正在处理退款，突然问“我的物流到哪了”，系统会中断退款任务：

```python
state.interrupted_active_task()
```

然后开启物流查询，同时激活中断提示流程：

```python
self._activate_interrupted_system_task(
    state,
    flow_list,
    interrupted_flow_id=interrupted_flow_id,
    interrupted_flow_name=interrupted_flow_name,
    started_flow_id=started_flow_id,
    started_flow_name=started_flow_name

```

对应回复类似：

```text
好的，我们先把退款申请放一放，先处理物流查询。
```

#### 3.6.3 恢复任务

用户说“继续刚才的”，可以恢复 paused_tasks 栈顶：

```python
if command.flow is not None:
    target_flow_id = command.flow
else:
    top_paused = state.paused_tasks[-1]
    target_flow_id = top_paused.flow_id
```

### 3.7 FlowExecutor 流程执行器

`FlowExecutor` 负责推进 YAML 中定义的流程。

核心入口：

```python
async def run_task(self,
                   state: DialogueState,
                   flows: FlowsList,
                   action_runner: ActionRunner):
    final_messages: list[BotMessage] = []
    while True:
        action_call: ActionCall = self._advance_until_action(state, flows)

        if action_call.action_name == "action_listen":
            break
        else:
            action_result: ActionResult = await action_runner.run(action_call, state)
            state.set_slots(action_result.slot_updates)
            final_messages.extend(action_result.messages)

    return final_messages
```

这个循环会一直推进流程，直到遇到 `action_listen`。

流程推进逻辑：

```python
current_active_task = state.current_active_task()

if current_active_task is None:
    return ActionCall(action_name="action_listen")

flow_id = current_active_task.flow_id
flow = flows.get_flow_by_id(flow_id)
step = flow.get_step_by_id(current_active_task.step_id)
action_call = self._run_step(state, step, flows)
```

不同 step 的处理方式：

| Step | 执行方式 |
| --- | --- |
| start | 推进到下一步 |
| collect | 检查槽位，缺失则激活系统收集流程 |
| action | 构造 `ActionCall` 返回给 ActionRunner |
| end | 清理 active_system_task 或 active_task |

#### 3.7.1 collect 槽位收集

```python
if state.active_task.slots.get(step.slot_name):
    self._advance_next_step(state, step)
    return None
else:
    state.start_active_system_task(CollectedSystemContext(
        flow_id="system_collect_information",
        step_id=flows.get_flow_by_id('system_collect_information').start_step().id,
        slot_name=step.slot_name,
        response=asdict(step.response)
    ))
    return None
```

如果槽位缺失，就进入 `system_collect_information`，让系统问用户：

```text
请告诉我你的订单号。
```

#### 3.7.2 对象消息填槽

如果用户点击了订单卡片或商品卡片，`focused_object` 可以自动填槽：

```python
if step.slot_name == 'order_number' and state.focused_object.type == "order":
    state.set_slots({step.slot_name: state.focused_object.id})
if step.slot_name == "product_id" and state.focused_object.type == "product":
    state.set_slots({step.slot_name: state.focused_object.id})
```

这就是前端对象消息和后端流程打通的关键。

### 3.8 Action 机制

Action 是真正执行业务动作的地方。

ActionResult 通常包含两类结果：

| 字段 | 说明 |
| --- | --- |
| `messages` | 要返回给用户的机器人消息 |
| `slot_updates` | 要写回当前任务 slots 的数据 |

#### 3.8.1 查询订单状态

```python
class LookUpOrderStatusAction(Action):
    name = "action_lookup_order_status"

    async def run(self, state: DialogueState, action_kwargs: dict[str, Any]) -> ActionResult:
        order_number = state.active_task.slots.get("order_number")
        payload = await fetch_order(order_number)

        if payload is None:
            return ActionResult(slot_updates={
                "order_status": "查询失败",
                "order_summary": "暂时无法查到该订单信息，请稍后再试。",
            })

        return ActionResult(slot_updates={
            "order_status": payload.get("status_desc") or payload.get("status") or "未知",
            "order_summary": _build_order_summary(payload),
        })
```

注意它不直接回复用户，而是写入槽位。

下一步 `action_response` 再把槽位渲染成话术：

```yaml
text: "订单{{ slots.order_number }}当前状态是：{{ slots.order_status }}。{{ slots.order_summary }}"
```

#### 3.8.2 查询物流

```python
return ActionResult(
    slot_updates={
        "tracking_number": logistics_data.get('tracking_number') or "未知",
        "logistics_company": logistics_data.get('logistics_company') or "未知",
        "logistics_status": logistics_data.get('status_desc') or logistics_data.get('status') or "未知"
    }
)
```

#### 3.8.3 ActionResponse

`action_response` 负责说话，支持三种模式：

| mode | 说明 |
| --- | --- |
| `static` | 直接模板渲染后返回 |
| `rephrase` | 先生成建议回复，再让 LLM 改写得更自然 |
| 其他 | 根据 prompt 直接生成回复 |

核心代码：

```python
mode = action_kwargs.get('mode', 'static')
text = action_kwargs['text']
render_text = self._render_text(state, text)

if mode == "static":
    return ActionResult(messages=[BotMessage(text=render_text)])
elif mode == "rephrase":
    prompt = action_kwargs['prompt']
    rephrase_text = await self._call_llm(state, prompt, render_text)
    return ActionResult(messages=[BotMessage(text=rephrase_text)])
```

模板渲染：

```python
template = Template(text)
result = template.render(
    slots=state.active_task.slots if state.active_task else {},
    context=state.active_system_task or state.active_task,
)
```

面试时可以强调：

> 查询类 action 负责拿数据、写槽位；response action 负责面向用户表达。这样业务查询和话术生成解耦，流程也更容易配置。

### 3.9 Knowledge 与 Chitchat 轨道

Knowledge 轨道处理信息咨询类问题。

知识意图定义：

```python
KNOWLEDGE_INTENTS: dict[str, KnowledgeIntent] = {
    "product_info": KnowledgeIntent(
        id="product_info", description="商品信息咨询",
        provider_ids=["api.product"], requires_object="product",
    ),
    "order_info": KnowledgeIntent(
        id="order_info", description="订单信息咨询",
        provider_ids=["api.order"], requires_object="order",
    ),
    "refund_policy": KnowledgeIntent(
        id="refund_policy", description="退款政策咨询",
        provider_ids=["faq.default", "rag.default"],
    ),
}
```

处理流程：

```python
provider_ids: list[str] = self._get_provider_ids_by_intents(knowledge_intents)

chunks: list[KnowledgeChunk] = []
for provider_id in provider_ids:
    provider = self.provider_registry.get(provider_id)
    current_chunks = await provider.retrieve(state)
    chunks.extend(current_chunks)

return await self.knowledge_responder.respond(
    user_message=state.pending_turn.user_message,
    recent_turns=state.current_session().turns[-5:],
    chunks=chunks
)
```

Knowledge 轨道有两个特点：

1. 通过 intent 找 provider
2. provider 返回 chunks 后，再由 LLM 组织答案

Chitchat 轨道更简单，主要把用户消息和最近历史交给 LLM，让它生成自然回复。

Clarify 则负责兜底，例如：

- LLM 没有命中任何轨道
- LLM 同时命中多个轨道
- task 命令非法
- knowledge 意图需要商品对象，但用户没有选择商品
- 用户点击对象但没有说明要查什么

面试回答：

> Knowledge、Chitchat、Clarify 都是为了把任务型客服流程外的情况处理掉。Task 轨道解决确定性业务，Knowledge 轨道解决知识问答，Chitchat 解决闲聊，Clarify 解决不确定性。

### 3.10 历史对话接口

历史接口：

```python
@router.get("/api/chat/history", response_model=ChatMessageResponse)
async def chat_history_endpoint(sender_id: str,
                                service: DialogueService = Depends(get_dialogue_service)) -> ChatMessageResponse:
    chat_message_response: list[ChatHistoryMessage] = await service.load_chat_history(sender_id)
    return ChatMessageResponse(sender_id=sender_id, messages=chat_message_response)
```

Service 层展开所有 session：

```python
async def load_chat_history(self, sender_id: str) -> list[ChatHistoryMessage]:
    state = await self.dialogue_repository.load(sender_id)

    chat_messages: list[ChatHistoryMessage] = []
    self._append_session_history_message(state.sessions, chat_messages)

    return chat_messages
```

历史消息模型：

```python
@dataclass(slots=True)
class ChatHistoryMessage:
    session_id: str
    role: Literal["user", "bot"]
    text: str | None = None
    object: FocusedObject | None = None
```

为什么返回 `session_id`？

- 它表示消息属于哪个后端业务 session
- 它保留后端会话结构，方便后续按 session 折叠、分组、排查问题
- 它不是用来判断前端分割线的

前端分割线逻辑是：

```text
页面打开时加载到的消息 = 历史消息
页面打开后用户新发的消息 = 当前消息
```

因此前端使用：

```js
const loadedHistoryCount = ref(0)
const currentPageDividerInserted = ref(false)
```

第一次发送新消息时，如果历史数量大于 0，就插入一次 divider。

面试时可以说：

> 后端只负责返回完整历史和 session 归属，不负责页面展示分割线。因为 current_session_id 是数据库里的当前业务 session，不等于浏览器本次打开页面后的新消息边界。分割线应该由前端根据页面生命周期判断。

### 3.11 数字人云渲染与语音播报

数字人这一块不是重新写一套客服系统，而是在已有对话能力外面加了一层“可视化播报通道”。核心思想是：**后端仍然复用 DialogueService 和 DialogueEngine 生成文本回复，前端把机器人文本交给数字人 SDK 播报**。

整体链路可以拆成两条：

```text
链路一：创建数字人会话
前端 App.vue
  ↓
GET /api/avatar/session?sender_id=u1001
  ↓
avatar_router.py
  ↓
infrastructure/avatar.py
  ↓
阿里云灵眸 CreateChatSession
  ↓
返回 sessionId + rtcParams
  ↓
前端 createAvatar(...) 初始化云渲染数字人
```

```text
链路二：数字人聊天播报
前端通过 WebSocket 发送 user_text
  ↓
/ws/avatar/chat
  ↓
avatar_ws_router.py
  ↓
DialogueService.handle_message
  ↓
DialogueEngine / Planner / Handler / Action
  ↓
返回 BotMessage 列表
  ↓
WebSocket 推送 bot_text
  ↓
前端 appendBotMessages 展示文本
  ↓
前端 avatar.requestToRespond(transcript, text)
  ↓
灵眸服务端完成 TTS + 唇形 + 数字人播报
```

也就是说，数字人没有改变原来的业务处理链路：

```text
原链路：用户文本 → DialogueService → BotMessage → 前端文本展示
数字人链路：用户文本 → DialogueService → BotMessage → 前端文本展示 + 数字人播报
```

#### 3.11.1 后端会话接口

数字人会话接口在 `avatar_router.py`：

```python
@router.get("/api/avatar/session")
async def create_avatar_session(
    sender_id: str | None = Query(default=None, description="可选: 透传作为 deviceId"),
) -> AvatarSessionResponse:
    data = await avatar.create_chat_session(device_id=sender_id)
    return AvatarSessionResponse(**data)
```

返回模型在 `schema.py`：

```python
class AvatarSessionResponse(BaseModel):
    sessionId: str
    rtcParams: dict = {}
    avatarAssets: dict = {}
```

这里的 `sessionId` 和 `rtcParams` 是前端 SDK 初始化云渲染数字人需要的参数，不是业务对话里的 `session_id`。这两个概念要区分：

| 概念                                  | 所属模块     | 作用                   |
| ------------------------------------- | ------------ | ---------------------- |
| `DialogueState.sessions[].session_id` | 客服对话状态 | 标识后端业务会话       |
| `AvatarSessionResponse.sessionId`     | 数字人云渲染 | 标识灵眸数字人渲染会话 |

面试时不要把这两个 session 混在一起讲。

#### 3.11.2 灵眸客户端封装

数字人客户端在 `infrastructure/avatar.py`，主要封装阿里云灵眸的 `CreateChatSession`：

```python
async def create_chat_session(device_id: str | None = None) -> dict[str, Any]:
    device = device_id or str(uuid.uuid4())
    payload = await asyncio.to_thread(_create_chat_session_sync, device)
    return payload
```

这里有一个细节：阿里云 SDK 调用是同步阻塞的，所以项目用：

```python
await asyncio.to_thread(...)
```

把阻塞调用放到线程池中，避免卡住 FastAPI 的异步事件循环。

配置项在 `.env` / `config.py` 中：

```text
AVATAR_ACCESS_KEY_ID
AVATAR_ACCESS_KEY_SECRET
AVATAR_PROJECT_ID
AVATAR_INSTANCE_ID
AVATAR_LICENSE
AVATAR_PLATFORM
AVATAR_ENDPOINT
```

应用启动时在 `app.py` 的 lifespan 中初始化数字人客户端：

```python
init_db_engine()
init_http_client()
init_dialogue_engine()
init_avatar_client()
```

这样数字人能力和数据库、HTTP 客户端、对话引擎一样，都是应用级基础设施。

#### 3.11.3 WebSocket 对话通道

数字人聊天使用 WebSocket：

```python
@router.websocket("/ws/avatar/chat")
async def avatar_chat_ws(ws: WebSocket) -> None:
    await ws.accept()
```

前端发送：

```json
{"type": "user_text", "sender_id": "u1001", "text": "我想查订单", "message_id": "..."}
```

后端返回：

```json
{"type": "user_ack", "message_id": "..."}
{"type": "bot_text", "message_id": "...", "index": 0, "text": "请提供订单号"}
{"type": "turn_end", "message_id": "..."}
```

这个 WebSocket 接口内部仍然复用原来的 service：

```python
result = await service.handle_message(user_message)
messages: list[BotMessage] = result.messages
```

所以数字人模式下，Planner、Validator、YAML flow、Action、Knowledge、Chitchat 全部复用，不需要再维护一套独立对话逻辑。

#### 3.11.4 打断机制

数字人播报有一个现实问题：如果上一轮还在播，用户又发了新消息，旧回复就不应该继续说完。

后端做法是保存当前 turn 的异步任务：

```python
current_turn: asyncio.Task | None = None
```

新消息到来时取消上一轮：

```python
await cancel_current("new user_text arrived")
await _safe_send_json(ws, {"type": "interrupt"})
current_turn = asyncio.create_task(_run_turn(ws, payload))
```

前端收到 `interrupt` 后：

```js
clearPendingPush()
clearTranscriptQueue()
avatarInstance.interrupt()
```

这样可以避免两个问题：

1. 旧回复和新回复同时往前端发
2. 数字人还在播旧内容，用户体验混乱

面试时可以说：

> 数字人播报是实时交互场景，所以必须支持打断。后端通过取消上一轮 asyncio task 停止旧回复继续发送，前端收到 interrupt 后停止播放器、清空播报队列，并调用 SDK 的 interrupt，让数字人立即停止当前播报。

#### 3.11.5 前端 SDK 接入

前端使用 `lm-avatar-chat-sdk`：

```js
import {
  createAvatar,
  TYAvatarType,
  TYVoiceChatMode,
  IAvatarAudioFormat,
  IRequestToRespondType,
} from 'lm-avatar-chat-sdk'
```

初始化数字人：

```js
const avatar = createAvatar(TYAvatarType.cloudAvatar, {
  rootContainer: '#cloudAvatarContainer',
  ...data.rtcParams,
  sessionId: data.sessionId,
  ignoreAudioInput: true,
})

avatar.start({
  mode: TYVoiceChatMode.tap2talk,
  avatarAudioFormat: IAvatarAudioFormat.s16,
  outboundSampleRate: 16000,
})
```

当前项目主要使用“文本复刻”模式：

```js
const USE_TRANSCRIPT_MODE = true
```

收到后端 `bot_text` 后，前端不是自己合成音频，而是把文本交给灵眸：

```js
transcriptQueue.push(payload.text)
flushTranscriptQueue()
```

真正触发播报的是：

```js
avatarInstance.requestToRespond(IRequestToRespondType.transcript, text)
```

这种模式下：

| 工作           | 谁负责              |
| -------------- | ------------------- |
| 业务对话理解   | 后端 DialogueEngine |
| 机器人文本生成 | 后端 BotMessage     |
| TTS 合成       | 灵眸服务端          |
| 唇形驱动       | 灵眸服务端          |
| 数字人渲染     | 前端 SDK + 云渲染   |

所以文档里讲数字人时，不能说“后端自己生成音频驱动数字人”。当前主链路是：**后端给文本，灵眸服务端负责 TTS 和唇形**。

#### 3.11.6 为什么保留音频驱动代码

前端代码里还保留了 `USE_TRANSCRIPT_MODE = false` 时的音频驱动逻辑：

```text
前端 CosyVoice TTS / PCM
  ↓
pushAudioData
  ↓
驱动数字人口型
```

这属于另一种模式，适用于“2D 云渲音频驱动”类型的灵眸项目。当前推荐使用 transcript 模式，因为链路更短：

```text
bot_text → requestToRespond → 灵眸服务端 TTS + 唇形
```

面试时可以这样回答：

> 我们最后采用的是文本复刻模式，而不是后端自己合成音频再推 PCM。原因是灵眸服务端本身支持根据 transcript 完成 TTS 和口型渲染，前端只需要把机器人文本交给 SDK。这样链路更简单，也减少了音频编码、采样率、播放器同步这些问题。

#### 3.11.7 数字人这一块面试怎么讲

可以按下面这段讲：

> 数字人模块没有改变客服系统的核心业务链路。用户文本通过 WebSocket 发到后端后，仍然走 DialogueService、DialogueEngine、Planner、Validator、YAML flow 和 Action，最后得到 BotMessage。区别是普通聊天接口直接把 BotMessage 返回给前端展示，而数字人模式通过 WebSocket 把 bot_text 推给前端，前端再调用 lm-avatar-chat-sdk 的 requestToRespond，把文本交给灵眸服务端做 TTS、唇形和云渲染。这样业务对话和数字人渲染是解耦的，后续即使换数字人厂商，也主要替换 avatar session 和前端 SDK 层，不影响核心对话引擎。

## 4. YAML 与 LangGraph 选型区别

这一章是面试里很容易被问到的点：**为什么不使用LangGraph，而使用 YAML 流程？**

### 4.1 流程特点不同

掌柜智库的导入和查询流程更像数据处理图：

```text
文档导入：PDF解析 → 图片处理 → 文档切分 → 商品识别 → 向量化 → Milvus写入
知识查询：商品确认 → 多路检索 → RRF融合 → 重排序 → 答案生成
```

它的特点是：

- 数据加工链路明显
- 查询流程可能多路并行
- 每个节点输入输出都是 graph state
- 适合用 LangGraph 表示节点和边

电商小二的业务流程更像客服 SOP：（标准作业程序）简单来说，它就是一份“傻瓜式说明书”。

```text
查订单：收集订单号 → 调订单接口 → 回复订单状态
查物流：收集订单号 → 调物流接口 → 回复物流进度
退款：收集订单号 → 收集退款原因 → 提交/回复结果
```

它的特点是：

- 流程确定性强
- 主要是 slot filling + action
- 需要业务人员能看懂流程
- 经常改话术、改步骤、改槽位
- 不需要复杂图计算或多路并行

所以电商小二选择 YAML 更合适。

### 4.2 YAML 流程的优势

| 优势 | 说明 |
| --- | --- |
| 可读性强 | 订单查询、退款申请这种 SOP 用 YAML 很直观 |
| 配置化 | 改流程不一定要改 Python 代码 |
| 适合业务沟通 | 产品、运营、测试也能看懂步骤 |
| 便于版本管理 | 流程配置可以像普通配置文件一样 review |
| 对 LLM 更安全 | LLM 只能选择已有 flow 和 command，不能自由编排任意节点 |
| 实现轻量 | 不引入图框架，方便教学和调试 |

一个典型流程：

```yaml
refund_request:
  steps:
    - type: collect
      slot_name: order_number
    - type: collect
      slot_name: refund_reason
    - type: action
      action: action_response
```

这对客服业务来说比代码图更直观。

### 4.3 LangGraph 的优势

LangGraph 更适合复杂 Agent / RAG / 数据处理工作流。

| 优势 | 说明 |
| --- | --- |
| 图结构表达能力强 | 适合复杂节点和条件边 |
| 并行能力更自然 | 多路检索、多个工具并发更方便 |
| 状态流转规范 | 每个节点读写 graph state |
| 适合复杂循环 | 反思、重试、工具调用循环更自然 |
| 生态能力强 | checkpoint、stream、可视化、debug 能力更完整 |

例如掌柜智库查询：

```text
商品名确认
    ↓
多路并行检索：向量检索 / HyDE / 联网搜索
    ↓
RRF 融合
    ↓
Reranker 重排序
    ↓
答案生成
```

这种流程有明显的图结构和并行节点，用 LangGraph 更自然。

### 4.4 为什么电商小二没有用 LangGraph

面试回答可以这样说：

> 电商小二的核心是客服业务流程，不是复杂 Agent 推理图。它的任务大多是确定性 SOP：收集槽位、调用业务接口、渲染回复、等待用户输入。我们希望流程能配置化、业务可读、容易修改，所以选择 YAML + 自研 FlowExecutor。LangGraph 更适合掌柜智库这种多节点、多路并行、检索融合的 RAG 工作流。如果强行用 LangGraph 做电商小二，会增加框架复杂度，但收益不明显。

### 4.5 选型对比表

| 维度 | YAML 编排 | LangGraph |
| --- | --- | --- |
| 适合场景 | 确定性业务流程、客服 SOP、槽位收集 | Agent 工作流、RAG 多路检索、复杂状态图 |
| 表达方式 | 配置文件描述 steps | Python 代码定义节点和边 |
| 业务可读性 | 高，非研发也容易理解 | 中等，需要理解图代码 |
| 并行能力 | 较弱，需要自研扩展 | 强，天然适合多节点并发 |
| 调试复杂度 | 简单，按 step 推进 | 更强但也更复杂 |
| 框架依赖 | 低 | 高 |
| 扩展复杂工作流 | 一般 | 强 |
| 本项目适配度 | 高 | 可以用，但偏重 |

### 4.6 如果面试官追问：以后会不会换 LangGraph

可以回答：

> 如果未来电商小二只是继续扩展查订单、查物流、退款、转人工等确定性流程，我会继续用 YAML。因为它更容易维护，也更适合业务配置。如果未来要加入复杂的多工具 Agent，比如并行查询多个系统、自动比较方案、循环规划和反思，或者要做复杂 RAG 检索链路，那可以把知识轨道或部分工具轨道迁移到 LangGraph，而不是把所有业务流程都替换掉。

也就是说，不是 YAML 和 LangGraph 二选一，而是：

```text
确定性客服流程：YAML
复杂检索/Agent 流程：LangGraph
```

## 5. 技术细节与痛点分析

### 5.1 如何降低 LLM 幻觉风险

项目没有让 LLM 直接执行接口调用，而是采用：

```text
LLM 输出 TurnPlan
    ↓
后端 Validator 校验
    ↓
CommandProcessor 修改状态
    ↓
FlowExecutor 执行 YAML
    ↓
Action 调业务接口
```

防护点包括：

- 只能命中一个轨道
- task command 必须在白名单里
- flow 必须在 YAML 中存在
- knowledge intent 必须在注册表中存在
- 需要对象的知识意图必须有 focused_object
- 校验失败走 clarify，而不是强行执行

### 5.2 为什么要设计系统流程

如果没有系统流程，任务切换和槽位收集就会散落在代码里。

例如：

```text
开始任务怎么说
中断任务怎么说
恢复任务怎么说
收集订单号怎么说
无法处理怎么说
```

这些都写在 `system_flows.yml` 里，可以复用 `action_response` 和 `action_listen`。

好处：

- 话术配置化
- 主流程更干净
- 任务切换逻辑统一
- 后续可以给系统提示加 rephrase 模式

### 5.3 为什么查询 action 不直接回复

以订单查询为例，`action_lookup_order_status` 只写槽位：

```python
return ActionResult(slot_updates={
    "order_status": payload.get("status_desc") or payload.get("status") or "未知",
    "order_summary": _build_order_summary(payload),
})
```

真正回复由下一步 `action_response` 完成。

这样做的好处：

- 查询和表达解耦
- 同一份查询结果可以被不同话术复用
- YAML 可以决定什么时候说、怎么说
- action 更容易测试

### 5.4 对象消息如何提升体验

前端可以传：

```json
{
  "sender_id": "u1001",
  "object": {
    "type": "order",
    "id": "ORDER_001",
    "title": "订单标题",
    "attributes": {}
  }
}
```

后端收到对象消息后：

```python
state.set_focused_object(user_message.object)
```

如果当前流程正在收集 `order_number`，就自动填入：

```python
if object_type == "order":
    if self._flow_has_unfilled_collect_slot(state, flows, "order_number"):
        return [SetSlotsCommand(command="set_slots", slots={"order_number": user_obj.id})]
```

这样用户不需要手输订单号，点击订单卡片就能继续流程。

### 5.5 当前设计的不足

| 问题 | 说明 | 可优化方向 |
| --- | --- | --- |
| YAML 条件使用 `eval` | 存在安全风险 | 换成安全表达式解析器 |
| 状态整体 JSON 存储 | 查询单条历史不方便 | 热路径 JSON，分析数据异步拆表 |
| 自研流程引擎 | 需要自己维护调试能力 | 增加流程可视化和执行日志 |
| Action 串行执行 | 不适合复杂并行工具调用 | 局部引入 LangGraph 或并发执行器 |
| YAML 校验较弱 | 配置错误可能运行时暴露 | 启动时做 Pydantic schema 校验 |
| FAQ/RAG Provider 简化 | 知识质量依赖后续实现 | 接入真实向量库和重排序 |

面试时不要只讲优点，也要主动讲这些取舍，会显得更真实。

## 6. 业务更换指南

这一章很重要。面试者写简历时，经常会把“电商客服”包装成“银行客服”“物流客服”“保险客服”“售后工单助手”等不同业务。如果只是改项目名称，不知道底层应该换哪些模块，面试时很容易被追问穿帮。

更换业务时不要理解成“把电商两个字替换掉”，而是要按下面这条链路系统替换：

```text
业务场景
  ↓
用户会问什么问题
  ↓
需要哪些业务流程 flow
  ↓
每个流程要收集哪些 slot
  ↓
每个流程要调用哪些外部系统 action
  ↓
哪些问题属于知识咨询 intent
  ↓
前端有没有业务对象卡片 object
  ↓
Planner prompt 里能不能让 LLM 选对 flow / intent
```

换业务时，底层对话框架大部分不用动，真正要动的是“业务配置 + 业务动作 + 业务知识 + 前端对象”。

### 6.1 先确定业务边界

第一步不是写代码，而是先把新业务拆清楚。

以“银行客服”为例，先列出用户常见问题：

| 用户问题         | 归类     | 后端处理方式                        |
| ---------------- | -------- | ----------------------------------- |
| 我要查银行卡余额 | 业务任务 | task flow + action 调余额接口       |
| 我要挂失银行卡   | 业务任务 | task flow + action 调挂失接口       |
| 信用卡怎么还款   | 知识咨询 | knowledge intent + FAQ/RAG provider |
| 你们周末上班吗   | 知识咨询 | knowledge intent + FAQ provider     |
| 谢谢你           | 闲聊     | chitchat                            |

这个表决定后面要改哪些 flow、slot、action 和 intent。

面试时可以说：

> 我更换业务时会先做意图盘点，把用户问题分成任务型、知识型和闲聊型。任务型进入 YAML flow，知识型进入 KnowledgeIntent 和 Provider，闲聊继续走 Chitchat，不会把所有问题都硬塞进一个流程。

### 6.2 修改业务流程 YAML

任务型问题要改：

```text
flow_config/user_flows.yml
```

原电商业务里有：

```yaml
order_status_query
logistics_tracking
refund_request
similar_product_recommendation
human_handoff
```

换成银行客服时，可以改成：

```yaml
balance_query:
  name: 余额查询
  description: 帮用户查询指定银行卡或账户的可用余额。
  steps:
    - id: start
      type: start
      next: ask_account_number

    - id: ask_account_number
      type: collect
      slot_name: account_number
      response:
        text: "请告诉我需要查询的银行卡号或账户号。"
      next: query_balance

    - id: query_balance
      type: action
      action: action_query_balance
      next: show_balance

    - id: show_balance
      type: action
      action: action_response
      args:
        text: "账户{{ slots.account_number }}当前可用余额为：{{ slots.available_balance }}。"
      next: end

    - id: end
      type: end
      next: []
```

再比如银行卡挂失：

```yaml
card_loss_report:
  name: 银行卡挂失
  description: 用户银行卡丢失后，收集银行卡号和身份信息，提交挂失申请。
  steps:
    - id: start
      type: start
      next: ask_card_number

    - id: ask_card_number
      type: collect
      slot_name: card_number
      response:
        text: "请告诉我需要挂失的银行卡号。"
      next: ask_identity_number

    - id: ask_identity_number
      type: collect
      slot_name: identity_number
      response:
        text: "为了核验身份，请提供身份证后四位。"
      next: report_card_loss

    - id: report_card_loss
      type: action
      action: action_report_card_loss
      next: show_result

    - id: show_result
      type: action
      action: action_response
      args:
        text: "银行卡{{ slots.card_number }}的挂失申请已提交，当前状态：{{ slots.loss_status }}。"
      next: end

    - id: end
      type: end
      next: []
```

这里要注意三点：

1. `flow_id` 要能表达业务含义，比如 `card_loss_report`
2. `description` 要写清楚，因为 Planner 会参考它判断用户意图
3. `action` 名称必须和 Python 里的 Action.name 对得上

### 6.3 修改 slot 定义

流程需要收集什么信息，就要在 YAML 顶部 `slots` 中定义什么字段。

电商里有：

```yaml
order_number
product_id
refund_reason
tracking_number
```

银行里可能换成：

```yaml
slots:
  account_number:
    type: text
    label: 账户号
    description: 用户要查询余额的银行账户号或银行卡号

  card_number:
    type: text
    label: 银行卡号
    description: 用户需要挂失或查询的银行卡号

  identity_number:
    type: text
    label: 身份证后四位
    description: 用于身份核验的身份证后四位

  available_balance:
    type: text
    label: 可用余额
    description: 查询到的账户可用余额

  loss_status:
    type: text
    label: 挂失状态
    description: 银行卡挂失申请的处理状态
```

slot 的作用不只是保存数据，还会影响 LLM 理解流程。

面试时可以强调：

> slot description 不是摆设。Planner 看到 flow 和 slot 描述后，才能判断用户说“查一下这张卡还有多少钱”时，应该开启 `balance_query`，并把卡号填入 `account_number` 或 `card_number`。

### 6.4 新增业务 Action

YAML 只描述流程，真正的业务动作要写成 Action。

电商项目里已有：

```text
atguigu/task/action/customer/lookup_order_status.py
atguigu/task/action/customer/lookup_logistics.py
atguigu/task/action/customer/recommend_similar_products.py
```

换银行业务时，可以新增：

```text
atguigu/task/action/customer/query_balance.py
atguigu/task/action/customer/report_card_loss.py
```

余额查询 Action 示例：

```python
from typing import Any
from atguigu.domain.state import DialogueState
from atguigu.task.action.base import Action, ActionResult
from atguigu.task.action.customer.bank_shared import fetch_balance


class QueryBalanceAction(Action):
    name = "action_query_balance"

    async def run(self, state: DialogueState, action_kwargs: dict[str, Any]) -> ActionResult:
        account_number = state.active_task.slots.get("account_number")
        payload = await fetch_balance(account_number)

        if payload is None:
            return ActionResult(slot_updates={
                "available_balance": "暂时无法查询，请稍后再试"
            })

        return ActionResult(slot_updates={
            "available_balance": payload.get("available_balance") or "未知"
        })
```

挂失 Action 示例：

```python
from typing import Any
from atguigu.domain.state import DialogueState
from atguigu.task.action.base import Action, ActionResult
from atguigu.task.action.customer.bank_shared import submit_card_loss


class ReportCardLossAction(Action):
    name = "action_report_card_loss"

    async def run(self, state: DialogueState, action_kwargs: dict[str, Any]) -> ActionResult:
        card_number = state.active_task.slots.get("card_number")
        identity_number = state.active_task.slots.get("identity_number")
        payload = await submit_card_loss(card_number, identity_number)

        if payload is None:
            return ActionResult(slot_updates={
                "loss_status": "提交失败，请稍后再试"
            })

        return ActionResult(slot_updates={
            "loss_status": payload.get("status") or "已受理"
        })
```

Action 写完后，要保证被注册到 action registry。项目里是通过 action builder / registry 统一管理 action，面试时可以表述为：

> 新增业务 action 后，需要让 ActionRegistry 能按 `action_query_balance` 找到对应 Action 类。YAML 中的 action 名称和 Python 类里的 `name` 必须完全一致，否则 FlowExecutor 构造 ActionCall 后，ActionRunner 找不到具体动作。

### 6.5 修改业务接口调用层

不要把外部 HTTP 请求散落在每个 action 里，建议像电商项目的 `shared.py` 一样抽公共模块。

电商里是：

```text
atguigu/task/action/customer/shared.py
```

银行可以改成：

```text
atguigu/task/action/customer/bank_shared.py
```

示例：

```python
from urllib.parse import quote
from atguigu.conf.config import settings
from atguigu.infrastructure.http_client import http_client


def _bank_base_url() -> str:
    return settings.bank_api_base_url.rstrip("/")


async def fetch_balance(account_number: str) -> dict | None:
    try:
        r = await http_client.get(f"{_bank_base_url()}/accounts/{quote(account_number)}/balance")
        payload = r.json()
        return payload.get("data") if isinstance(payload, dict) else None
    except Exception:
        return None


async def submit_card_loss(card_number: str, identity_number: str) -> dict | None:
    try:
        r = await http_client.post(
            f"{_bank_base_url()}/cards/{quote(card_number)}/loss-report",
            json={"identity_number": identity_number}
        )
        payload = r.json()
        return payload.get("data") if isinstance(payload, dict) else None
    except Exception:
        return None
```

配置也要同步换：

```text
atguigu/conf/config.py
.env
```

比如：

```env
BANK_API_BASE_URL=http://localhost:9000
```

面试时可以说：

> 更换业务系统时，我不会让 Action 直接拼一堆 HTTP 细节，而是抽一个 shared client 层。Action 只表达业务动作，HTTP URL、异常处理、响应解析放到共享模块里。

### 6.6 修改知识意图和 Provider

不是所有用户问题都应该写成 task flow。

比如银行客服里：

| 用户问题         | 是否需要流程 | 处理方式         |
| ---------------- | ------------ | ---------------- |
| 我要挂失银行卡   | 需要         | task flow        |
| 信用卡怎么还款   | 不需要       | knowledge intent |
| 跨行转账多久到账 | 不需要       | knowledge intent |
| 我这张卡余额多少 | 需要         | task flow        |

知识意图定义在：

```text
atguigu/knowledge/intents.py
```

银行场景可以改成：

```python
"credit_card_repayment_policy": KnowledgeIntent(
    id="credit_card_repayment_policy",
    description="信用卡还款规则咨询",
    provider_ids=["faq.bank", "rag.bank"],
),
"transfer_arrival_policy": KnowledgeIntent(
    id="transfer_arrival_policy",
    description="转账到账时间咨询",
    provider_ids=["faq.bank", "rag.bank"],
),
"account_info": KnowledgeIntent(
    id="account_info",
    description="账户信息咨询",
    provider_ids=["api.account"],
    requires_object="account",
),
```

Provider 要同步调整：

```text
atguigu/knowledge/provider.py
atguigu/knowledge/registry.py
```

例如：

```python
class BankFAQProvider(KnowledgeProvider):
    id = "faq.bank"

    async def retrieve(self, state: DialogueState) -> list[KnowledgeChunk]:
        return [
            KnowledgeChunk(content="信用卡支持自动还款、主动还款和柜台还款。"),
        ]
```

如果简历里写“银行客服助手”，面试官问“信用卡还款规则怎么回答”，就不能说走 `refund_policy`，而应该说走银行业务自己的 `credit_card_repayment_policy`。

### 6.7 修改对象消息和前端卡片

电商项目支持前端点击订单卡片、商品卡片：

```text
order → order_number
product → product_id
```

换业务后，如果前端也有业务对象，就要同步修改对象类型。

例如银行客服可能有：

```text
account → account_number
card → card_number
loan → loan_id
```

后端需要改对象填槽逻辑，位置在：

```text
atguigu/engine/dialogue_engine.py
```

原来是：

```python
if object_type == "order":
    if self._flow_has_unfilled_collect_slot(state, flows, "order_number"):
        return [SetSlotsCommand(command="set_slots", slots={"order_number": user_obj.id})]

if object_type == "product":
    if self._flow_has_unfilled_collect_slot(state, flows, "product_id"):
        return [SetSlotsCommand(command="set_slots", slots={"product_id": user_obj.id})]
```

银行可以改成：

```python
if object_type == "account":
    if self._flow_has_unfilled_collect_slot(state, flows, "account_number"):
        return [SetSlotsCommand(command="set_slots", slots={"account_number": user_obj.id})]

if object_type == "card":
    if self._flow_has_unfilled_collect_slot(state, flows, "card_number"):
        return [SetSlotsCommand(command="set_slots", slots={"card_number": user_obj.id})]
```

前端也要同步：

```text
订单卡片 / 商品卡片 → 银行账户卡片 / 银行卡片
object.type: order/product → account/card
object.id: order_id/product_id → account_number/card_number
```

面试时可以说：

> 对象消息是前后端协作点。换业务时不只改后端 flow，还要把前端卡片对象类型、后端 FocusedObject 类型、slot 自动填充规则一起替换，否则用户点击卡片后无法自动填槽。

### 6.8 修改 Planner 可理解的信息

Planner 会根据当前可用 flows、slots、knowledge intents、历史上下文来判断用户意图。

所以换业务时，不一定要大改 planner 代码，但必须保证这些描述足够准确：

| 信息                         | 要求                           |
| ---------------------------- | ------------------------------ |
| flow name                    | 能表达业务动作，如“银行卡挂失” |
| flow description             | 写清楚适用场景和用户说法       |
| slot description             | 写清楚槽位含义                 |
| knowledge intent description | 写清楚知识问题范围             |

例如不要写：

```yaml
description: 处理用户问题
```

要写：

```yaml
description: 用户银行卡丢失或被盗刷时，收集银行卡号和身份信息，提交挂失申请。
```

面试时可以说：

> 我们的 Planner 是通用的，业务迁移时重点是把 flow、slot、intent 的自然语言描述写准确。LLM 不是靠硬编码 if else 判断业务，而是根据这些描述选择结构化 TurnPlan。

## 7. 面试高频问题与回答

### 7.1 你这个项目整体是怎么工作的？

回答：

> 用户请求进入 FastAPI 后，先转成 UserMessage。Service 根据 sender_id 从数据库加载 DialogueState，然后交给 DialogueEngine。Engine 会准备 session、创建 pending_turn，再根据消息类型处理。文本消息先让 LLM Planner 输出结构化 TurnPlan，再经过 Validator 校验，最后分发到 Task、Knowledge 或 Chitchat 轨道；对象消息则用于 focused_object 和槽位填充。处理完成后，bot 消息写入当前 turn，整个 DialogueState 再持久化到数据库。

### 7.2 为什么要有 DialogueState？

回答：

> 因为客服是多轮对话，不能只看当前一句话。DialogueState 里保存 active_task、paused_tasks、active_system_task、focused_object、sessions、current_session_id 等信息。这样系统才能知道用户正在进行哪个流程、已经填了哪些槽位、是否有被中断的任务、历史上下文是什么。

### 7.3 LLM 在项目里具体做什么？

回答：

> LLM 主要**做理解和生成**，不直接执行业务。第一类是 TurnPlanner，根据用户消息和上下文判断走 task、knowledge 还是 chitchat，并输出结构化命令。第二类是 response rephrase 或 knowledge responder，用来把建议回复或知识片段组织成自然语言。真正的状态修改、流程推进和接口调用都由后端代码完成。

### 7.4 如何防止 LLM 乱输出？

回答：

> 我们用了 TurnPlanValidator。它会校验是否只命中一条轨道、task 命令是否属于白名单、是否有多个 start_flow、flow 是否存在、knowledge intent 是否存在，以及需要商品或订单对象时 focused_object 是否满足。校验失败时不执行，而是走 ClarifyResponder 追问用户。

### 7.5 YAML 流程是怎么跑起来的？

回答：

> YAML 里定义了 flow 和 steps，每个 step 有 type、id、next。LLM 通过 start_flow 命令激活某个业务流程，CommandProcessor 把 active_task 设置为该 flow 的 start step。之后 FlowExecutor 不断读取当前 step：start 直接推进，collect 检查槽位，action 构造 ActionCall，end 结束任务。遇到 action_listen 时停止，等待下一轮用户输入。

### 7.6 为什么不用 LangGraph，而是自己写 YAML 流程？

回答：

> 因为电商客服流程更像确定性的业务 SOP，比如收集订单号、查订单、回复状态。它强调配置化、业务可读、容易调整。YAML 能让产品和测试也看懂流程，而且 LLM 只能选择已有 flow，安全边界更清楚。LangGraph 更适合复杂 Agent 或 RAG 图，比如多路检索、并行节点、反思循环。这个项目如果用 LangGraph 也能做，但会增加框架复杂度，收益不明显。

### 7.7 那 LangGraph 什么时候更合适？

回答：

> 如果流程有复杂节点依赖、并行执行、多轮工具调用、检索融合、循环重试，LangGraph 更合适。比如掌柜智库查询流程里有商品确认、向量检索、HyDE、联网搜索、RRF、重排序、答案生成，这种天然是图结构。电商小二的 task 流程主要是线性的槽位收集和 action 执行，所以 YAML 更合适。

### 7.8 task、knowledge、chitchat 怎么区分？

回答：

> task 是需要执行业务流程的问题，比如查订单、查物流、退款。knowledge 是信息咨询，比如退货政策、商品信息、平台规则。chitchat 是闲聊寒暄。LLM Planner 根据当前用户消息、历史、active_task、focused_object、可用 flows 和 intents 输出轨道，然后 Validator 保证只选一个轨道。

### 7.9 如何支持任务中断和恢复？

回答：

> DialogueState 里有 active_task 和 paused_tasks。用户开启新任务时，如果当前已经有 active_task，CommandProcessor 会把旧任务压入 paused_tasks，然后开启新任务，并触发 system_task_interrupted 提示。用户说继续刚才的，ResumeFlowCommand 会从 paused_tasks 栈顶恢复任务，或者按指定 flow_id 恢复。

### 7.10 对象消息是怎么处理的？

回答：

> 前端点击订单或商品卡片时，会发送 object 消息。后端把它转成 FocusedObject 存到 state.focused_object。如果当前流程正在收集 order_number 或 product_id，Engine 会把对象 id 转成 SetSlotsCommand，自动填入槽位。如果没有明确任务，就走 clarify，让用户说明想基于这个对象做什么。

### 7.11 ActionResult 为什么分 messages 和 slot_updates？

回答：

> 因为有些 action 是查数据，不负责说话，比如查询订单只把 order_status、order_summary 写入 slots。后续 action_response 再根据 YAML 模板渲染回复。messages 表示直接返回给用户的话，slot_updates 表示更新任务上下文的数据。这样查询和表达解耦。

### 7.12 历史对话为什么返回 session_id？

回答：

> session_id 表示这条消息属于哪个后端业务 session。历史接口把所有 session 的 turn 展平成消息列表，如果不返回 session_id，就丢失了后端会话归属信息。它不是用来判断前端分割线的，前端分割线是根据页面打开时的历史快照和本页第一次新消息判断的。

### 7.13 数字人模块是怎么接入的？

回答：

> 数字人模块不是单独的一套客服逻辑，而是在原有对话链路外面加了一层播报通道。前端先调用 `/api/avatar/session` 获取灵眸云渲染需要的 `sessionId` 和 `rtcParams`，再用 `lm-avatar-chat-sdk` 初始化数字人。用户文本通过 `/ws/avatar/chat` 发给后端，后端仍然复用 `DialogueService.handle_message`，拿到 `BotMessage` 后通过 WebSocket 推送 `bot_text`，前端再调用 `requestToRespond(transcript, text)` 让数字人播报。

### 7.14 数字人里的 sessionId 和历史对话里的 session_id 是一回事吗？

回答：

> 不是。历史对话里的 `session_id` 是 DialogueState 中的业务会话 ID，用来表示一条消息属于哪个客服 session。数字人接口返回的 `sessionId` 是灵眸云渲染会话 ID，用来初始化前端数字人 SDK。前者服务于业务对话状态，后者服务于音视频云渲染，不能混在一起。

### 7.15 数字人播报为什么用 WebSocket？

回答：

> 数字人聊天有实时性要求，不只是一次 HTTP 请求返回文本。WebSocket 可以持续推送 `user_ack`、`bot_text`、`interrupt`、`turn_end` 等事件。比如用户新发消息时，服务端可以取消上一轮任务并推送 `interrupt`，前端收到后停止播放器、清空播报队列，并调用数字人 SDK 的 `interrupt`，避免旧回复继续播。

### 7.16  状态整体 JSON 存储有什么优缺点？

回答：

> 优点是实现简单，恢复上下文方便，active_task、paused_tasks、slots、focused_object、sessions 都能一次性恢复，适合多轮对话状态。缺点是不好按 turn 查询、统计和分析，每次更新也是整份 JSON 更新。生产环境可以保留 JSON 作为在线状态，同时异步拆出对话明细表做分析。

### 7.1.7 对话状态膨胀怎么办？

**问题描述**：
长时间对话后，DialogueState JSON 变得非常大（积累大量 Turn 历史）。

**解决方案 - 会话裁剪**：

```python
MAX_TURNS_PER_SESSION = 20      # 每个 Session 最多 20 轮
MAX_SESSIONS = 5                 # 最多保留 5 个 Session
MAX_COMPLETED_TASKS = 10         # 最多保留 10 个已完成任务

def trim_state(state: DialogueState) -> DialogueState:
    # 1. 裁剪每个 Session 的 Turns
    for session in state.sessions:
        if len(session.turns) > MAX_TURNS_PER_SESSION:
            session.turns = session.turns[-MAX_TURNS_PER_SESSION:]

    # 2. 裁剪 Session 数量
    if len(state.sessions) > MAX_SESSIONS:
        state.sessions = state.sessions[-MAX_SESSIONS:]

    # 3. 裁剪已完成任务
    if len(state.completed_tasks) > MAX_COMPLETED_TASKS:
        state.completed_tasks = state.completed_tasks[-MAX_COMPLETED_TASKS:]

    return state
```

### 7.17 这个项目有哪些可以优化的地方？

回答：

> 第一，YAML 条件现在用 eval，可以换成安全表达式解析器。第二，YAML 可以加 Pydantic 校验，在启动时发现配置错误。第三，FlowExecutor 可以加执行 trace，方便排查流程推进。第四，历史状态可以拆出明细表用于分析。第五，知识轨道可以接入真实 RAG、重排序和引用来源。第六，如果未来出现复杂并行工具调用，可以在局部引入 LangGraph。

#### 【30 秒版】

> 我参与开发了一个电商智能客服系统，核心是“LLM 负责理解，后端负责执行”。系统采用 task、knowledge、chitchat 三轨路由，由 LLM Planner 判断用户意图，再由 Validator 校验结构化 TurnPlan。确定性业务任务通过 YAML flow 编排，由 CommandProcessor、FlowExecutor 和 ActionRunner 执行，支持查订单、查物流、退款、商品推荐、转人工、知识咨询和闲聊。同时系统支持多轮状态持久化、任务中断恢复、对象卡片填槽、历史对话展示，以及数字人云渲染播报。

#### 【2 分钟版】

> **项目背景**
>
> 这是一个面向电商平台的智能客服系统，目标是替代传统关键词匹配客服。用户可以用自然语言咨询订单、物流、退款、商品和平台规则，也可以直接点击订单或商品卡片继续对话。
>
> **技术架构**
>
> 系统分为四层：
>
> 1. **前端层**：Vue 3 聊天界面，支持订单/商品对象卡片、历史消息分割线和数字人播报。
> 2. **客服后端**：FastAPI + LangChain，负责 DialogueState、多轮会话、LLM Planner、Validator、YAML Flow、Action 和 WebSocket 数字人通道。
> 3. **业务后端**：模拟电商订单、物流、商品等业务 API。
> 4. **存储层**：MySQL，按用户保存完整 DialogueState JSON，支持断点续聊和历史对话恢复。
>
> 核心流程是：用户消息进入 FastAPI 后，先加载 DialogueState，再由 DialogueEngine 调用 TurnPlanner 做三轨判断，Validator 校验后分发到 Task、Knowledge 或 Chitchat。Task 轨道会通过 CommandProcessor 修改任务状态，再由 FlowExecutor 推进 YAML 流程，最后 Action 调业务接口或生成回复。
>
> **我的职责**
>
> 我主要负责 DialogueEngine 调度、三轨路由、Command 状态机、YAML 流程编排、Action 机制、历史对话展示，以及数字人 WebSocket 播报链路。
>
> **技术亮点**
>
> 第一，LLM 不直接执行业务，只输出结构化计划，后端用 Validator 和 CommandProcessor 保证状态一致性。第二，YAML flow 把业务 SOP 配置化，便于扩展和讲解。第三，active_task + paused_tasks 支持任务中断恢复。第四，对象消息可以把订单或商品 ID 自动填入 slot，降低 LLM 抽取错误。第五，数字人模式复用同一套 DialogueService，只是把 bot_text 通过 WebSocket 推给前端 SDK 播报。

#### 【5 分钟版】

> **一、项目背景与目标**
>
> 电商客服是高频场景，传统方案依赖关键词匹配和固定问答树，用户表达稍微变化就容易无法命中。这个项目的目标是用 LLM 理解自然语言，同时用后端流程引擎保证业务执行可控，让系统既能自然对话，又能完成查订单、查物流、退款等确定性任务。
>
> **二、整体架构设计**
>
> 系统采用前后端分离和多服务协作：
>
> - 前端：Vue 3 聊天页面，支持文本消息、对象卡片、历史分割线和数字人播报。
> - 客服后端：FastAPI，负责对话状态、三轨路由、流程编排、Action 执行和历史接口。
> - 业务后端：提供订单、物流、商品等模拟电商接口。
> - 存储：MySQL 保存 DialogueState JSON。
>
> 三轨路由是核心设计：Task 轨处理多步骤业务流程，Knowledge 轨处理商品、订单、退款政策等知识咨询，Chitchat 轨处理闲聊寒暄。三条轨道共享 DialogueState，因此系统能利用历史上下文、当前任务、已选对象和已填槽位。
>
> **三、核心技术实现**
>
> **TurnPlan + Validator**：LLM Planner 不直接回复用户，而是输出结构化 TurnPlan。Validator 会校验是否只命中一个轨道、flow 是否存在、command 是否合法、knowledge intent 是否存在、是否缺少 focused_object。校验失败不会强行执行，而是进入 clarify 追问。
>
> **Command 状态机**：Planner 输出 start_flow、set_slots、resume_flow、cancel_flow 等命令，CommandProcessor 负责真正修改 DialogueState。这样 LLM 的不确定性被限制在计划层，状态变化由确定性代码完成。
>
> **YAML 流程编排**：查订单、查物流、退款等业务流程写在 `flow_config/user_flows.yml` 中，每个流程由 start、collect、action、end 组成。FlowExecutor 根据当前 step 推进流程，遇到 action_listen 就停下来等待下一轮用户输入。
>
> **任务中断与恢复**：DialogueState 中有 active_task 和 paused_tasks。用户在退款流程中突然查物流时，系统会把退款任务压入 paused_tasks，再开启物流任务；用户说“继续刚才的”，就从栈顶恢复任务。
>
> **对象感知消息**：前端点击订单或商品卡片时，会发送 object 消息。后端把它转成 FocusedObject，如果当前流程正在收集 order_number 或 product_id，就自动填槽，减少用户输入和 LLM 抽取错误。
>
> **历史对话展示**：后端 `/api/chat/history` 返回所有 session 的历史消息，并给每条消息带 `session_id`。后端不返回 divider，前端根据页面首次加载快照和本页第一次新消息插入“以上为历史消息”分割线。
>
> **数字人播报**：数字人模块没有重写对话逻辑。前端先调用 `/api/avatar/session` 获取灵眸云渲染的 sessionId 和 rtcParams，再通过 `/ws/avatar/chat` 发送用户文本。后端复用 DialogueService 生成 BotMessage，并把 bot_text 推给前端，前端调用 `lm-avatar-chat-sdk` 的 `requestToRespond`，由灵眸服务端完成 TTS、唇形和云渲染。
>
> **四、遇到的挑战与解决方案**
>
> 最大挑战是 LLM 输出不可控。我的方案是让 LLM 只做“规划”，不直接改状态、不直接调接口；再用 Validator 做结构校验，用 CommandProcessor 做状态变更，用 YAML flow 和 ActionRunner 做确定性执行。
>
> 第二个挑战是多任务切换。通过 active_task + paused_tasks 的 LIFO 栈设计，用户可以自然地从退款切到物流，再恢复退款。
>
> 第三个挑战是历史消息分割线。如果后端按 current_session_id 切分，用户刚打开页面就可能看到奇怪的分割线。所以我把后端职责收敛为返回完整历史和 session 归属，页面生命周期边界交给前端判断。
>
> 第四个挑战是数字人播报打断。用户新发消息时，后端取消上一轮 asyncio task 并推送 interrupt，前端停止播放器、清空队列并调用 SDK interrupt，避免旧内容继续播。
>
> **五、项目成果与收获**
>
> 项目实现了完整的电商客服闭环，覆盖任务型客服、知识咨询、闲聊兜底、对象消息、任务中断恢复、历史对话和数字人展示。技术上我加深了对 LLM 应用边界、状态机、多轮对话、流程编排和前后端实时通信的理解。
>
> 如果重新优化，我会优先补充 YAML schema 校验、流程执行 trace、真实 RAG 知识库、SSE/流式回复，以及数字人失败时的降级策略。

### D. STAR 法则回答模板

**问题**：请介绍一个你解决过的技术难题。

```text
S（Situation，情境）
在开发电商客服系统时，我们发现如果直接让 LLM 决定回复和业务动作，风险很高。
LLM 可能输出不合法 JSON，也可能选择不存在的流程，甚至在任务切换时破坏当前对话状态。

T（Task，任务）
我需要设计一套机制，让系统既能利用 LLM 的自然语言理解能力，
又能保证业务流程、状态变更和接口调用是稳定可控的。

A（Action，行动）
我把 LLM 的职责限制在“输出结构化 TurnPlan”。
然后设计了三层防护：

1. TurnPlanValidator：校验轨道是否唯一、flow 是否存在、command 是否合法、intent 是否存在。
2. CommandProcessor：只允许 start_flow、set_slots、resume_flow、cancel_flow 等白名单命令修改状态。
3. FlowExecutor + YAML：真正的业务流程由后端确定性执行，LLM 不能自由调用任意接口。

同时，校验失败时不强行执行，而是进入 ClarifyResponder，让系统追问用户。

R（Result，结果）
这样系统既保留了 LLM 的语义理解能力，又避免了它直接影响核心业务状态。
后续新增查物流、退款、商品推荐等流程时，只需要扩展 YAML 和 Action，主流程不需要大改。
```

另一个可以讲的 STAR 例子是“历史分割线设计”：

```text
S：后端原来按 current_session_id 区分历史和当前消息，用户刚进入页面就可能看到分割线。
T：需要实现“首次加载历史不显示分割线，首次发送新消息才显示分割线”的体验。
A：我把后端改成只返回所有历史消息和 session_id，不返回 divider；前端用 loadedHistoryCount 和 currentPageDividerInserted 判断页面生命周期边界。
R：职责更清晰，后端保留业务会话归属，前端负责展示边界，避免了初始页面分割线异常。
```

### E. 常见追问

| 追问                                             | 应答要点                                                     |
| ------------------------------------------------ | ------------------------------------------------------------ |
| 为什么不用 LangGraph，而是 YAML？                | 电商客服流程偏确定性 SOP，主要是槽位收集和 action 执行，YAML 更轻、更直观；LangGraph 更适合复杂 RAG、多路并行和 Agent 图。 |
| LLM 为什么不直接生成最终回复？                   | 业务系统不能完全相信 LLM。LLM 只负责规划，后端用 Validator、CommandProcessor、FlowExecutor 保证执行可控。 |
| Command 状态机和传统状态机有什么区别？           | 传统状态机转换规则完全写死；这里由 LLM 输出结构化 Command 表达意图，但最终状态变更仍由后端白名单命令执行。 |
| LIFO 栈为什么不用 FIFO？                         | 用户切换任务通常是“临时处理新问题，再回到刚才的问题”，后进先出更符合对话直觉。 |
| 为什么 DialogueState 整体 JSON 存储？            | 恢复上下文简单，active_task、paused_tasks、slots、sessions 能一次恢复；缺点是分析查询不方便，生产可异步拆明细表。 |
| 为什么历史接口返回 session_id？                  | 保留后端业务 session 归属，方便分组、折叠、排查；前端分割线不是靠 session_id，而是靠页面生命周期判断。 |
| 为什么历史分割线不放后端？                       | 后端不知道用户本次页面打开的边界，current_session_id 只是数据库业务 session，分割线是前端展示概念。 |
| 对象消息有什么价值？                             | 用户点击订单/商品卡片后，后端可直接拿 object.id 填 slot，减少手输和 LLM 抽取错误。 |
| ActionResult 为什么分 messages 和 slot_updates？ | 查询类 action 只负责拿数据写槽位，回复由 action_response 渲染，实现查询和表达解耦。 |
| 数字人模块会不会影响原对话链路？                 | 不影响。数字人只是展示层增强，仍复用 DialogueService 和 DialogueEngine，只是通过 WebSocket 把 bot_text 推给前端 SDK 播报。 |
| 数字人的 sessionId 和业务 session_id 一样吗？    | 不一样。`session_id` 是客服业务会话，`sessionId` 是灵眸云渲染会话。 |
| 如果数字人服务失败怎么办？                       | 可以降级为普通文本聊天；核心对话链路不依赖数字人云渲染。     |
| 如何处理用户模糊意图？                           | Validator 或 Planner 不确定时走 ClarifyResponder，向用户追问，而不是强行选择流程。 |
| 项目还能怎么优化？                               | YAML schema 校验、流程 trace、真实 RAG、SSE 流式响应、数字人降级策略、局部 LangGraph。 |
