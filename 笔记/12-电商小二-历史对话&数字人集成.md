# 数字人集成

---

## 第 1 章 这一节做什么

前一节已经把 Action 和 Handler 的实现补齐了。到这里，客服系统在“文字对话”层面已经能跑通：用户输入一句话，后端可以完成意图判断、任务流转、查订单、查物流、知识问答和闲聊回复。

这一节要做的是把文字客服升级成“看得见、听得到”的数字人客服。用户还是在前端输入文字，后端仍然复用同一套 `DialogueService` 和 `DialogueEngine`，区别只是：机器人回复出来以后，不只显示在聊天气泡里，还要交给阿里云灵眸数字人，由数字人把这段话说出来。

本节的目标可以先压成一句话：

> 用户输入 → 后端对话引擎生成回复 → WebSocket 推送回复文本 → 前端调用灵眸 SDK → 阿里云完成 TTS、唇形和云渲染 → 数字人开口说话。

这里要特别注意一个边界：**当前项目使用 transcript 模式**。也就是说，后端只生成和推送回复文本，不做 TTS，也不推 PCM 音频。声音、唇形、画面渲染都交给阿里云灵眸服务端完成。

**PCM** 的全称是 **Pulse-Code Modulation（脉冲编码调制）**。简单来说，它是**将自然界中连续的模拟音频信号（比如人说话的声音）转换成计算机能够理解和处理的离散数字信号**的标准方法。在开发语音合成（TTS）、自动语音识别（ASR）或实时对话的底层系统时，PCM 几乎是一定会接触到的最核心、最基础的音频数据格式。



本节会交付这些内容：

| 交付件 | 位置 | 作用 |
| --- | --- | --- |
| 阿里云资源开通说明 | 第 3 章 | 准备 AccessKey、ProjectId、InstanceId、License 等配置 |
| 完整交互时序图 | 第 4 章 | 把前端、后端、对话引擎、阿里云之间的调用关系讲清楚 |
| 后端依赖和配置 | 第 5 章 | 加灵眸 SDK 依赖和 `.env` 配置字段 |
| 后端会话客户端 | 第 6 章 | 封装 `CreateChatSession`、全局缓存和释放 |
| HTTP 会话路由 | 第 7 章 | 给前端提供数字人会话参数 |
| WebSocket 对话路由 | 第 8 章 | 推送 `bot_text`，支持打断和一轮对话收尾 |
| 前端核心接入 | 第 9 章 | 初始化 SDK、打开 WS、调用 `requestToRespond` |
| 历史对话展示 | 第 10 章 | 历史接口、前后端分工、分割线交互流程 |

有了这个总目标，下一章先看整体方案，否则直接进入代码会很容易把 HTTP、WebSocket、RTC 三条连接混在一起。

## 第 2 章 总体架构

数字人集成不是简单多调一个接口，而是同时引入三条链路：

| 链路 | 方向 | 承载内容 | 为什么需要 |
| --- | --- | --- | --- |
| HTTP | 前端 → 后端 → 阿里云 | 数字人会话参数 | 前端 SDK 初始化前必须先拿到 `sessionId` 和 `rtcParams` |
| WebSocket | 前端 ↔ 后端 | 用户文本、机器人文本、打断事件 | 对话过程需要低延迟双向通信 |
| RTC | 前端 SDK ↔ 阿里云灵眸 | 数字人音视频流 | 数字人的画面和声音由阿里云云端渲染后推到浏览器 |

先看一张带顺序编号的 ASCII 时序图：

![](images/1、总体架构.png)

这张图里最容易混淆的是“声音从哪里来”。当前项目不是后端合成声音，也不是浏览器本地 TTS，而是：

1. 后端只生成 `BotMessage.text`
2. WebSocket 把 `bot_text` 发给前端
3. 前端调用 `avatar.requestToRespond(IRequestToRespondType.transcript, text)`
4. 阿里云灵眸服务端把文字转成声音，并生成对应唇形
5. 浏览器通过 RTC 收到数字人的音视频流

所以，本节的后端重点不是音频处理，而是**会话创建、会话释放、对话文本推送和打断控制**。

> 1. **HTTP 会话和 WebSocket 对话是两条不同链路**。HTTP 只负责拿 SDK 初始化参数；真正一问一答走 WebSocket。
> 2. **WebSocket 只推文本**。`bot_text` 到前端以后，前端再通过 SDK 让灵眸服务端完成 TTS 和渲染。



实现顺序按依赖关系来：

| 顺序 | 做什么 | 关键文件 |
| --- | --- | --- |
| 1 | 准备阿里云资源 | 阿里云控制台 / `.env` |
| 2 | 增加依赖和配置字段 | `pyproject.toml` / `atguigu/config/config.py` |
| 3 | 封装灵眸会话客户端 | `atguigu/infrastructure/avatar.py` |
| 4 | 暴露 HTTP 会话接口 | `atguigu/api/router/avatar_router.py` |
| 5 | 暴露 WebSocket 对话接口 | `atguigu/api/router/avatar_ws_router.py` |
| 6 | 注册路由和初始化客户端 | `atguigu/api/app.py` |
| 7 | 前端接入 SDK 和 WS | `customer-service-frontend/src/App.vue` |
| 8 | 配置 Vite 代理 | `customer-service-frontend/vite.config.js` |

后面章节会按这个顺序展开。先从最基础也最容易出错的阿里云资源开始。

## 第 3 章 阿里云资源开通

代码写得再对，只要阿里云资源缺一个字段，前端都拿不到数字人会话。所以这一章先把需要开通和记录的资源讲清楚。

### 3.1 最终要拿到哪些字段

最终要写进 `customer-service-backend/.env` 的数字人字段如下：

| 字段 | 来自哪里 | 用途 |
| --- | --- | --- |
| `AVATAR_ACCESS_KEY_ID` | RAM 用户 AccessKey | 后端调用灵眸 OpenAPI 的身份凭证 |
| `AVATAR_ACCESS_KEY_SECRET` | RAM 用户 AccessKey | 后端调用灵眸 OpenAPI 的密钥 |
| `AVATAR_ENDPOINT` | 灵眸服务地域 | 指定 OpenAPI 接入点，当前项目默认北京地域 |
| `AVATAR_PROJECT_ID` | 灵眸项目详情 | `CreateChatSession` 的 `project_id` |
| `AVATAR_INSTANCE_ID` | 灵眸实例详情 | 指定使用哪个数字人实例 |
| `AVATAR_LICENSE` | 灵眸实例详情 | Web 端授权字段 |
| `AVATAR_PLATFORM` | 项目约定 | 当前写 `web` |

如果对话引擎本身还要调用 Qwen，大模型部分仍然要保留已有配置：

```bash
LLM_MODEL=qwen-plus
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_API_KEY=sk-********
```

数字人字段负责“让小雨开口”，LLM 字段负责“让后端生成回复”，这两组配置不要混在一起理解。

### 3.2 创建 RAM AccessKey

灵眸 OpenAPI 使用阿里云通用的 AccessKey 鉴权，所以第一步是创建一个 RAM 用户。

推荐流程：

1. 进入阿里云 RAM 访问控制
2. 创建用户
3. 访问方式选择 OpenAPI 调用访问
4. 创建后记录 `AccessKey ID` 和 `AccessKey Secret`
5. 给这个 RAM 用户授予调用灵眸相关 OpenAPI 的权限

这里不要用主账号 AccessKey。主账号 AK 泄露的影响面太大，课程环境可以先用较宽权限跑通，生产环境应收敛到最小权限。

拿到以后先记录成：

```bash
AVATAR_ACCESS_KEY_ID=LTAI********
AVATAR_ACCESS_KEY_SECRET=********
```

### 3.3 开通万相数字人 / 灵眸服务

第二步是在阿里云控制台开通万相数字人，也就是灵眸服务。开通后要确认服务地域，因为地域决定 OpenAPI endpoint。

当前代码默认使用北京地域：

```bash
AVATAR_ENDPOINT=lingmou.cn-beijing.aliyuncs.com
```

如果你在控制台选择了其他地域，就要按对应地域修改 `AVATAR_ENDPOINT`。Endpoint 写错时，后端通常会在第一次 `CreateChatSession` 时失败。

### 3.4 创建项目：选择 2D Chat

进入灵眸控制台后，需要先创建项目。这里有一个关键选择：**当前项目要选 2D Chat / 对话类项目，不要选音频驱动项目。**

两类项目的区别如下：

| 项目类型 | 前端 SDK 调用 | TTS 在哪里做 | 当前项目是否使用 |
| --- | --- | --- | --- |
| 2D Chat / 对话类 | `requestToRespond(transcript, text)` | 灵眸服务端 | 使用 |
| 2D 音频驱动 | `pushAudioData(int16, end)` | 自己的后端或浏览器 | 不使用 |

当前前端代码中有一个开关：

```javascript
const USE_TRANSCRIPT_MODE = true
```

这个 `true` 就对应 2D Chat 项目。项目类型如果选错，最常见的现象是：接口都不报错，但数字人就是不开口。

项目创建完成后，在项目详情里记录项目 ID：

```bash
AVATAR_PROJECT_ID=********
```

### 3.5 创建实例：选择形象、声音和模型

项目下面还要创建实例。实例可以理解为“具体是哪一个数字人”：形象、声音、基础模型、画面比例等都在实例里配置。

创建实例时重点关注四项：

| 配置 | 说明 |
| --- | --- |
| 形象 | 选择预置 2D 形象，或者使用自己上传 / 定制的形象 |
| 声音 | 选择预置音色，或者绑定克隆音色 |
| 模型 | 灵眸内部可挂 LLM，但当前项目直接把后端回复文本传给数字人，所以这里选默认即可 |
| 画面比例 | 根据前端展示区域选择，课程项目一般 3:4 或 16:9 都能调 |

实例创建后，需要记录：

```bash
AVATAR_INSTANCE_ID=avatar_2dchat_public_cn-xxxxxxxxxx
AVATAR_LICENSE=********
AVATAR_PLATFORM=web
```

其中 `AVATAR_PLATFORM` 是我们自己的约定字段，当前 Web 前端固定填 `web`。

### 3.6 .env 模板

最后把字段统一放到 `customer-service-backend/.env`：

### 3.7 常见配置错误

| 现象 | 优先检查 |
| --- | --- |
| `Avatar AccessKey 未配置` | `.env` 是否在 `customer-service-backend` 目录，字段名是否是大写 |
| `Avatar 项目 ID 未配置` | 是否填写了 `AVATAR_PROJECT_ID` |
| `CreateChatSession failed` | AK/SK 权限、Endpoint、ProjectId、InstanceId 是否匹配同一账号和地域 |

## 第4章 端点和消息协议

先看 HTTP 端点：

| 方法 | 路径 |  | 返回 | 用途 |
| --- | --- | --- | --- | --- |
| GET | `/api/avatar/session` | `sender_id?` | `{sessionId, rtcParams, avatarAssets}` | 创建或复用当前进程的数字人会话 |
| DELETE | `/api/avatar/session` | 无 | `{released: boolean}` | 释放当前进程缓存的数字人会话 |
| WS | `/ws/avatar/chat` | WebSocket | JSON 文本帧 | 对话文本、打断、收尾 |

WebSocket 协议如下：

| 方向 | 类型 | 示例 | 说明 |
| --- | --- | --- | --- |
| 前端 → 后端 | `user_text` | `{"type":"user_text","sender_id":"u1001","text":"查订单","message_id":"m1"}` | 用户发一条新消息 |
| 后端 → 前端 | `user_ack` | `{"type":"user_ack","message_id":"m1"}` | 后端确认收到 |
| 后端 → 前端 | `bot_text` | `{"type":"bot_text","message_id":"m1","index":0,"text":"..."}` | 一条机器人回复 |
| 后端 → 前端 | `turn_end` | `{"type":"turn_end","message_id":"m1"}` | 本轮回复结束 |
| 后端 → 前端 | `interrupt` | `{"type":"interrupt"}` | 要求前端停止当前播报 |
| 后端 → 前端 | `error` | `{"type":"error","message":"..."}` | 异常 |

### 5.2 Settings新增数字人配置

`atguigu/config/config.py` 中增加这一组字段：

```python
from pathlib import Path
from pydantic_settings import BaseSettings, SettingsConfigDict

ENV_FILE_DIR = Path(__file__).resolve().parents[2]
ENV_FILE_PATH = ENV_FILE_DIR / ".env"


class Settings(BaseSettings):
    model_config=SettingsConfigDict(env_file=ENV_FILE_PATH, env_file_encoding="utf-8", extra='ignore')

    llm_model: str
    llm_base_url: str
    llm_api_key: str
    commerce_api_base_url: str
    database_url: str
    app_host: str
    app_port: int

    # 万相数字人(灵眸) 云渲染. 说话由灵眸服务端 TTS 完成, 后端不再做 TTS
    avatar_access_key_id: str = ""
    avatar_access_key_secret: str = ""
    avatar_endpoint: str = "lingmou.cn-beijing.aliyuncs.com"
    avatar_project_id: str = ""
    avatar_license: str = ""
    avatar_platform: str = "web"
    avatar_instance_id: str = ""


settings = Settings()
```

这里有三个设计点：

1. `extra='ignore'`：`.env` 里多出来的字段不会让应用启动失败。
2. Avatar 字段都有默认值：没填数字人配置时，应用仍然能启动，只是在调用数字人接口时才报错。
3. `.env` 用大写字段，代码里用小写属性：`AVATAR_PROJECT_ID` 会被 `pydantic-settings` 映射到 `settings.avatar_project_id`。

## 第 6 章 后端会话客户端

`atguigu/infrastructure/avatar.py` 是数字人后端实现的核心。它只做一件事：把阿里云灵眸的阻塞式 SDK 调用，封装成 FastAPI 可以安全使用的异步函数。

它要解决三个实际问题：

| 问题 | 处理方式 |
| --- | --- |
| 灵眸 Python SDK 是同步阻塞调用 | 用 `asyncio.to_thread` 放到线程池执行 |
| 公共实例并发位有限 | 后端进程内全局复用同一个数字人 session |
| 会话不再使用时需要归还并发位 | 通过 `DELETE /api/avatar/session` 主动释放当前缓存的 session |

完整代码如下：

```python
"""阿里云万相数字人(灵眸)会话客户端: 封装 CreateChatSession。

供前端 `lm-avatar-chat-sdk` 初始化云渲染数字人使用。
SDK 调用是阻塞的, 通过 `asyncio.to_thread` 包装成异步接口供 FastAPI 使用。

"""
from __future__ import annotations

import logging
import uuid
import asyncio
from typing import Any

from alibabacloud_lingmou20250527.client import Client as LingMouClient
from alibabacloud_lingmou20250527 import models as lm_models
from alibabacloud_tea_openapi import models as open_api_models
from alibabacloud_tea_util import models as util_models

from atguigu.config.config import settings

logger = logging.getLogger(__name__)

_client: LingMouClient | None = None


def init_avatar_client() -> None:
    """在 FastAPI lifespan 中调用, 提前完成客户端初始化。"""
    global _client
    if _client is not None:
        return
    if not settings.avatar_access_key_id or not settings.avatar_access_key_secret:
        logger.warning(
            "Avatar AccessKey is not configured; lazy init will fail on first request."
        )
        return
    _client = _build_client()
    logger.info("Avatar client initialized. endpoint=%s", settings.avatar_endpoint)


def _build_client() -> LingMouClient:
    config = open_api_models.Config(
        access_key_id=settings.avatar_access_key_id,
        access_key_secret=settings.avatar_access_key_secret,
    )
    config.endpoint = settings.avatar_endpoint
    return LingMouClient(config)


def _get_client() -> LingMouClient:
    global _client
    if _client is None:
        if not settings.avatar_access_key_id or not settings.avatar_access_key_secret:
            raise RuntimeError(
                "Avatar AccessKey 未配置, 请在 .env 设置 "
                "AVATAR_ACCESS_KEY_ID / AVATAR_ACCESS_KEY_SECRET"
            )
        _client = _build_client()
    return _client


def _rtc_params_to_dict(rtc) -> dict[str, Any]:
    if rtc is None:
        return {}
    return {
        "appId": rtc.app_id,
        "channel": rtc.channel,
        "nonce": rtc.nonce,
        "timestamp": rtc.timestamp,
        "token": rtc.token,
        "gslb": rtc.gslb,
        "clientUserId": rtc.client_user_id,
        "serverUserId": rtc.server_user_id,
        "avatarUserId": rtc.avatar_user_id,
    }



def _do_create(device_id: str) -> dict[str, Any]:
    """对应官方 demo:
    client.create_chat_session_with_options(project_id, request, headers, runtime)
    project_id 是控制台项目主键(如 'C1rRS1KmS3WurHor8HXYlSkQ');
    request 里通常只填 instance_id, 其余 device_id 由调用方按需补。
    """
    client = _get_client()
    project_id = settings.avatar_project_id
    if not project_id:
        raise RuntimeError("Avatar 项目 ID 未配置, 请在 .env 设置 AVATAR_PROJECT_ID")

    req_kwargs: dict[str, Any] = {}
    if settings.avatar_instance_id:
        req_kwargs["instance_id"] = settings.avatar_instance_id
    if device_id:
        req_kwargs["device_id"] = device_id
    if settings.avatar_license:
        req_kwargs["license"] = settings.avatar_license
    if settings.avatar_platform:
        req_kwargs["platform"] = settings.avatar_platform
    req = lm_models.CreateChatSessionRequest(**req_kwargs)

    resp = client.create_chat_session_with_options(
        project_id, req, {}, util_models.RuntimeOptions()
    )
    data = getattr(resp.body, "data", None)
    if data is None:
        raise RuntimeError(
            f"CreateChatSession returned empty data. request_id={getattr(resp.body, 'request_id', None)}"
        )
    return {
        "sessionId": data.session_id,
        "rtcParams": _rtc_params_to_dict(data.rtc_params),
    }


def _create_chat_session_sync(device_id: str) -> dict[str, Any]:
    return _do_create(device_id)


async def create_chat_session(device_id: str | None = None) -> dict[str, Any]:
    """创建一次数字人云渲染会话, 返回前端 SDK 初始化所需 JSON。"""
    device = device_id or str(uuid.uuid4())

    payload = await asyncio.to_thread(_create_chat_session_sync, device)
    return payload

```

### 6.1 客户端初始化

`init_avatar_client()` 在应用启动时调用，但它找不到 AK/SK 时不会抛异常，只打印 warning。

这样设计是为了让没有数字人凭证的开发者仍然可以启动后端，继续调试普通文本对话接口。真正访问 `/api/avatar/session` 时，`_get_client()` 才会检查 AK/SK 并抛出清晰错误。

### 6.2 响应字段转换

阿里云 SDK 返回的是 Python 对象，而且字段名偏 snake_case。前端 SDK 需要的是 camelCase JSON，所以这里做了两层转换：

| 函数 | 输入 | 输出 |
| --- | --- | --- |
| `_rtc_params_to_dict` | `data.rtc_params` | `appId`、`channel`、`token`、`gslb` 等 |
| `_avatar_assets_to_dict` | `data.avatar_assets` | `type`、`url`、`md5`、`secret` 等 |

最终返回给前端的结构固定为：

```json
{

  "rtcParams": {
    "appId": "...",
    "channel": "...",
    "nonce": "...",
    "timestamp": "...",
    "token": "...",
    "gslb": [],
    "clientUserId": "...",
    "serverUserId": "...",
    "avatarUserId": "..."
  },
  "avatarAssets": {
    "type": "...",
    "url": "...",

    "secret": "...",
    "minRequiredVersion": "..."
  }

```

前端拿到以后可以直接展开给 `createAvatar` 使用。

### 6.3 CreateChatSession调用

`_do_create()` 是真正调用阿里云的地方：

```python
    project_id, req, {}, util_models.RuntimeOptions()

```

### 6.4 `_session_lock` 的作用

`_create_chat_session_sync()` 和 `_stop_chat_session_sync()` 都会读写模块级变量 `_session`，所以必须用 `_session_lock` 把“检查、创建、替换、清空”这些操作包成原子流程。

如果没有这把锁，两个请求同时进入 `GET /api/avatar/session` 时，可能都看到 `_session is None`，从而同时调用阿里云创建两个数字人 session，浪费并发位。加锁后，第一个请求创建并写入 `_session`，第二个请求拿到锁时会直接复用已经存在的 session。

释放时也一样。`DELETE /api/avatar/session` 会先在锁内取出并清空 `_session`，再调用阿里云关闭 session。这样可以避免释放过程中其他线程读到一个“正在关闭”的旧 session；如果关闭失败，再把原 session 放回缓存，后续还能继续重试释放。

## 第 7 章 HTTP 会话路由

HTTP 路由负责让前端拿到数字人 SDK 初始化需要的参数。它不处理聊天内容，只处理数字人会话本身。

### 7.1 响应模型

先在 `atguigu/api/schema.py` 中增加 `AvatarSessionResponse`：

```python
class AvatarSessionResponse(BaseModel):
    """前端 lm-avatar-chat-sdk 初始化所需会话 JSON。"""
    sessionId: str
    rtcParams: dict = {}
    avatarAssets: dict = {}
```

字段名使用 camelCase，是为了和前端 SDK 接收的结构保持一致。

### 7.2 路由完整代码

`atguigu/api/router/avatar_router.py` 完整代码如下：

```python
"""数字人云渲染相关 HTTP 路由。"""
from fastapi import APIRouter, Query

from atguigu.api.schemas import AvatarSessionResponse
from atguigu.infrastructure import avatar

router = APIRouter()


@router.get("/api/avatar/session")
async def create_avatar_session(
    sender_id: str | None = Query(default=None, description="业务用户 ID, 数字人会话会全局复用"),
) -> AvatarSessionResponse:
    """创建数字人云渲染会话, 给前端 SDK 初始化。"""
    data = await avatar.create_chat_session(device_id=sender_id)
    return AvatarSessionResponse(**data)


@router.delete("/api/avatar/session")
async def release_avatar_session() -> dict:
    """释放当前数字人会话, 归还并发位。"""
    released = await avatar.stop_chat_session()
    return {"released": released}

```

这个路由有两个边界要分清：

1. `sender_id` 是业务对话身份，不是数字人会话缓存 key。后端进程内只缓存一个数字人 session。
2. `DELETE /api/avatar/session` 不接收 `sender_id`，释放的是当前进程缓存的全局数字人 session。

HTTP 会话只解决“数字人上线”。下一章讲“用户说一句，后端回一句”的 WebSocket 路由。

## 第 8 章 WebSocket 对话路由

前面第 7 章已经解决了“数字人怎么上线”：前端通过 `GET /api/avatar/session` 拿到 `sessionId` 和 `rtcParams`，再交给灵眸 SDK 建立 RTC 连接。

但是，数字人上线以后还不会自动回答问题。真正的聊天内容要走另一条链路：前端把用户输入通过 WebSocket 发给后端，后端复用现有 `DialogueService` 得到机器人回复，再把回复文本推回前端。

所以第 8 章只讲后端这条文本通道：

```text
前端 user_text
    -> /ws/avatar/chat
    -> DialogueService.hand_dialogue()
    -> bot_text / turn_end
    -> 前端再交给数字人播报
```

这里要先明确一个边界：**WebSocket 只传 JSON 文本帧，不传 PCM 音频帧**。当前项目使用 transcript 模式，后端只返回文字；TTS、唇形和云渲染由前端调用灵眸 SDK 后交给阿里云完成。

### 8.1 为什么需要 WebSocket

普通 `POST /api/chat` 是一次请求对应一次响应：

```text
前端发请求 -> 后端处理 -> 后端 return -> 请求结束
```

数字人聊天更适合 WebSocket，因为它需要一条持续存在的“电话线”：

```text
前端先连接 /ws/avatar/chat
连接不断开时，前端可以随时发送 user_text
后端也可以随时推送 user_ack、bot_text、interrupt、turn_end
```

这样做有两个好处：

| 能力 | 为什么 HTTP 不够方便 |
| --- | --- |
| 主动推送 | 后端可以把多条 `BotMessage` 逐条推给前端 |
| 打断上一轮 | 用户新输入到来时，后端可以取消旧任务，并通知前端停播 |

### 8.2 WebSocket 消息协议

当前协议只有 JSON 文本消息：

| 方向 | 类型 | 示例 | 说明 |
| --- | --- | --- | --- |
| 前端 -> 后端 | `user_text` | `{"type":"user_text","sender_id":"u1001","text":"查订单","message_id":"m1"}` | 用户发起一轮文本对话 |
| 后端 -> 前端 | `user_ack` | `{"type":"user_ack","message_id":"m1"}` | 后端确认收到这一轮 |
| 后端 -> 前端 | `bot_text` | `{"type":"bot_text","message_id":"m1","index":0,"text":"..."}` | 一条机器人回复文本 |
| 后端 -> 前端 | `turn_end` | `{"type":"turn_end","message_id":"m1"}` | 本轮回复全部发送完成 |
| 后端 -> 前端 | `interrupt` | `{"type":"interrupt"}` | 要求前端停止旧播报、清空旧队列 |
| 后端 -> 前端 | `error` | `{"type":"error","message":"..."}` | 边界输入或处理失败 |

一轮正常对话的顺序是：

```text
前端 -> user_text
后端 -> interrupt
后端 -> user_ack
后端 -> bot_text 0
后端 -> bot_text 1
后端 -> turn_end
```

这里的 `interrupt` 看起来有点奇怪：为什么第一条消息也会发？因为后端逻辑统一处理“新 user_text 到来”这个事件：先取消可能存在的上一轮，再通知前端停掉旧播报。第一轮没有旧任务时，它只是一个无害的停播通知。

### 8.3 事件类型如何前后端配合

WebSocket 里没有 HTTP 那种固定的 request / response 结构。前端和后端只是沿着同一条连接互相发送事件，所以要靠 `type` 字段约定每条消息的含义。

一次完整交互可以按下面理解：

```text
1. 前端发送 user_text
   表示用户发起一轮新问题。

2. 后端发送 interrupt
   表示旧轮次作废，前端要停止数字人当前播报，并清空还没播的文本队列。

3. 后端发送 user_ack
   表示后端已经接受这一轮消息，message_id 已经进入后端处理流程。

4. 后端发送一条或多条 bot_text
   每条 bot_text 都是一条机器人回复。前端收到后，一边显示聊天气泡，一边放入 transcriptQueue 等待数字人播报。

5. 后端发送 turn_end
   表示这一轮 bot_text 已全部发送完成。前端收到后结束 pendingTurn，让发送按钮恢复可用。

6. 任意阶段后端都可能发送 error
   表示输入不合法或后端处理失败。前端收到后展示错误，并结束当前 pendingTurn。
```

对应到当前前端 `handleSocketJson()` 的处理关系是：

| 后端事件 | 前端处理 | 用户看到的效果 |
| --- | --- | --- |
| `user_ack` | 暂不展示，只保留协议确认 | 页面无明显变化 |
| `bot_text` | `appendBotMessages()` + `transcriptQueue.push()` + `flushTranscriptQueue()` | 出现机器人气泡，数字人开始排队播报 |
| `turn_end` | `pendingTurn.resolve()`，清空当前等待轮次 | 本轮发送完成，可以继续发送下一条 |
| `interrupt` | `pcmPlayer.stop()`、`clearTranscriptQueue()`、`avatar.interrupt()` | 旧播报停止，新问题优先 |
| `error` | 设置 `errorMessage`，`rejectPendingTurn()` | 页面显示错误，当前轮次结束 |

所以这几个事件的职责可以压缩成一句话：

```text
user_ack 管接收确认，bot_text 管回复内容，turn_end 管轮次结束，interrupt 管打断旧播报，error 管失败收尾。
```

后端只负责把这些事件按顺序推出来；前端负责把事件翻译成 UI 状态、队列状态和数字人 SDK 调用。

### 8.4 路由主体：连接循环

`avatar_ws_router.py` 里的入口是：

```python
@router.websocket("/ws/avatar/chat")
async def avatar_chat_ws(ws: WebSocket) -> None:
    await ws.accept()
    current_turn: asyncio.Task | None = None
```

`await ws.accept()` 表示接起这条 WebSocket 连接。后面的 `current_turn` 表示“当前正在处理的那一轮用户问题”。

主体循环如下：

```python
try:
    while True:
        try:
            payload = json.loads(await ws.receive_text())
        except json.JSONDecodeError:
            await _send_json(ws, {"type": "error", "message": "invalid json"})
            continue

        if payload.get("type") != "user_text":
            await _send_json(
                ws, {"type": "error", "message": f"unknown type: {payload.get('type')}"}
            )
            continue

        await cancel_current("new user_text arrived")
        await _send_json(ws, {"type": "interrupt"})
        current_turn = asyncio.create_task(_run_turn(ws, payload))
except WebSocketDisconnect:
    logger.info("WS client disconnected")
finally:
    await cancel_current("ws closing")
```

这段代码可以拆成四步：

| 步骤 | 作用 |
| --- | --- |
| `while True` | WebSocket 是长连接，所以要持续等待下一条消息 |
| `json.loads(await ws.receive_text())` | 收到前端发来的 JSON 文本帧 |
| `payload.get("type")` | 只接受当前协议支持的 `user_text` |
| `create_task(_run_turn(...))` | 启动一轮对话处理，同时让外层循环继续接收新消息 |

这里的 `except WebSocketDisconnect` 会在浏览器关闭页面、刷新页面、网络断开，或者前端主动关闭 WebSocket 时触发。它不是业务错误，只说明连接正常结束，所以后端只记一条日志。真正的清理放在 `finally` 里统一执行，确保连接断开时也会取消当前还没处理完的对话任务。

为什么不直接 `await _run_turn(ws, payload)`？

因为对话引擎可能要访问数据库、调用大模型、执行任务动作。如果直接 `await`，这段时间里外层循环就不能及时收到下一条 `user_text`，也就无法打断旧回答。用 `create_task` 后，外层循环继续保持接听状态；一旦新消息进来，就可以取消旧任务。

### 8.5 cancel_current 的作用

`cancel_current()` 不是为了“兜底”，而是为了支持数字人聊天里的打断体验。

```python
async def cancel_current(reason: str) -> None:
    nonlocal current_turn
    if current_turn is None or current_turn.done():
        current_turn = None
        return

    current_turn.cancel()
    try:
        await current_turn
    except (asyncio.CancelledError, Exception):
        pass
    finally:
        current_turn = None
    logger.info("WS turn cancelled: %s", reason)
```

它解决的是这种场景：

```text
用户问 A
后端正在处理 A，或者前端正在播 A 的回复
用户又问 B
```

如果没有取消逻辑，A 和 B 的回复可能混在一起推给前端，数字人就会串话。现在的策略是：**新消息来了，旧轮次作废**。

这里有两个细节：

| 代码 | 含义 |
| --- | --- |
| `current_turn.cancel()` | 通知旧任务停止 |
| `await current_turn` | 等旧任务真的退出，避免旧任务继续发送 `bot_text` |

`except (asyncio.CancelledError, Exception)` 只作用在等待旧任务退出这一步。`CancelledError` 表示旧任务响应了取消请求，属于预期结果；`Exception` 表示旧任务在退出过程中自己已经失败。这里统一忽略，是因为 `cancel_current()` 的目标不是处理旧任务业务结果，而是确保旧任务不再继续往当前 WebSocket 发送消息。无论旧任务是被取消还是异常结束，最后都要把 `current_turn` 清空。

### 8.6 _run_turn：复用现有对话引擎

`_run_turn()` 才是真正处理一轮用户输入的函数。

第一步，从前端 payload 取字段并组装领域对象：

```python
sender_id = payload.get("sender_id")
text = payload.get("text") or ""
message_id = payload.get("message_id") or str(uuid.uuid4())

user_message = UserMessage(
    sender_id=sender_id,
    message_id=message_id,
    type=MessageType.TEXT,
    text=text,
)
```

`sender_id` 和 `text` 是边界输入，所以这里保留校验：缺少 `sender_id` 或空文本时返回 `error`。

第二步，先回一个 ACK：

```python
await _send_json(ws, {"type": "user_ack", "message_id": message_id})
```

`user_ack` 不代表机器人已经回答完成，只代表后端收到了这一轮请求。前端当前不展示它，但它可以让协议更完整。

第三步，复用现有服务层：

```python
async with db.session_factory() as session:
    repository = DialogueRepository(session=session)
    service = DialogueService(repository=repository, engine=get_engine())
    result = await service.hand_dialogue(user_message)
messages: list[BotMessage] = result.messages
```

这一步是整个设计的关键：数字人没有新写一套业务逻辑，仍然走原来的 `DialogueService` 和 `DialogueEngine`。数字人链路只是把输出方式从 HTTP response 改成 WebSocket push。

| 普通文字聊天 | 数字人聊天 |
| --- | --- |
| `POST /api/chat` | `/ws/avatar/chat` |
| 返回 `ChatResponse` | 推送 `bot_text` |
| 前端显示气泡 | 前端显示气泡，并交给数字人播报 |

第四步，把机器人回复逐条推给前端：

```python
for index, bot_msg in enumerate(messages):
    await _send_json(
        ws,
        {
            "type": "bot_text",
            "message_id": message_id,
            "index": index,
            "text": bot_msg.text,
            "object": _focused_object_to_dict(bot_msg.object),
        },
    )

await _send_json(ws, {"type": "turn_end", "message_id": message_id})
```

如果一轮对话返回多条 `BotMessage`，前端会收到多条 `bot_text`，最后用 `turn_end` 判断这一轮结束。

`_run_turn()` 里的 `except asyncio.CancelledError: raise` 专门处理“这一轮被外层取消”的情况。它通常发生在两种时候：用户在上一轮还没结束时又发了新消息，或者 WebSocket 连接关闭触发了 `finally` 清理。这里必须重新 `raise`，让取消信号继续往上传给 `cancel_current()`；如果把它吞掉，外层会以为旧任务正常结束，取消语义就不清晰，也容易把任务取消和普通业务异常混在一起。

普通业务异常走后面的 `except Exception`。例如数据库访问失败、对话引擎报错、消息处理异常等，后端会记录日志并向前端发送 `error` 事件，让当前轮次失败收尾。

### 8.7 注册 WebSocket 路由

WebSocket 路由写完后，还要挂到 FastAPI 应用上。当前应用入口是 `atguigu/api/app.py`：


## 第 9 章 前端数字人接入

第 8 章讲的是后端如何接收 `user_text` 并推回 `bot_text`。但 `bot_text` 只是文字，数字人并不会因为前端收到一段文字就自动开口。

```text
拿数字人会话参数

    -> 收到 bot_text

    -> 阿里云完成 TTS + 唇形 + RTC 渲染
```

前端相关代码集中在 `customer-service-frontend/src/App.vue`，开发代理在 `customer-service-frontend/vite.config.js`。

### 9.1 核心状态和 transcript 模式

前端引入灵眸 SDK：

```javascript
import {
  createAvatar,
  TYAvatarType,
  TYVoiceChatMode,
  IAvatarAudioFormat,
  IRequestToRespondType,
} from 'lm-avatar-chat-sdk'

const USE_TRANSCRIPT_MODE = true
```

`USE_TRANSCRIPT_MODE = true` 是当前项目最重要的前后端约定：

| 模式 | 前端做什么 | 后端做什么 | 当前是否使用 |
| --- | --- | --- | --- |
| transcript | 调 `requestToRespond(transcript, text)` | 只推送 `bot_text` | 使用 |
| audio driven | 调 `pushAudioData(int16, end)` | 需要推 PCM 音频帧 | 不使用 |

所以当前真实链路里，`handleSocketBinary()`、`pcmPlayer`、`pushAudioData()` 都不是主路径。主路径只有 `bot_text -> transcriptQueue -> requestToRespond()`。

数字人相关状态主要有三组：

```javascript
const avatarState = ref('idle')
const wsState = ref('idle')
let avatarInstance = null
let avatarReady = false
let chatSocket = null
let pendingTurn = null

let transcriptQueue = []
let transcriptInFlight = false
let transcriptSawActiveState = false

```

| 状态 | 作用 |
| --- | --- |
| `avatarState` | 页面展示用，表示数字人画面是否 ready |
| `wsState` | 页面展示用，表示 WebSocket 是否连接 |

| `chatSocket` | 连接后端 `/ws/avatar/chat` 的 WebSocket |
| `pendingTurn` | 当前发送出去但还没收到 `turn_end` 的一轮消息 |
| `transcriptQueue` | 等待交给数字人播报的回复文本队列 |

### 9.2 页面加载：先准备数字人

页面挂载时会执行：

```javascript
onMounted(async () => {
  initBg()
  window.addEventListener('beforeunload', handleBeforeUnload)
  ensureSetupAfterGesture()
  await Promise.all([fetchSidebarData(), fetchChatHistory()])
})
```

这里有两条互不阻塞的初始化线：

| 初始化 | 作用 |
| --- | --- |
| `ensureSetupAfterGesture()` | 准备数字人 RTC 画面和 SDK |
| `fetchChatHistory()` | 加载历史聊天记录，给第 10 章使用 |

为什么不是直接调用 `setupAvatar()`？因为浏览器有自动播放策略，音频能力通常需要用户手势后才能启用。所以前端先检查：

```javascript
function ensureSetupAfterGesture() {
  if (avatarInstance || pendingSetupAfterGesture) return
  if (hasUserActivation()) {
    setupAvatar()
    return
  }
  pendingSetupAfterGesture = true
  avatarState.value = 'waiting-gesture'
  window.addEventListener('pointerdown', trigger, true)
  window.addEventListener('keydown', trigger, true)
}
```

如果浏览器已经有用户激活，直接初始化数字人；否则页面显示“待启用”，等用户点击或按键后再调用 `setupAvatar()`。

### 9.3 setupAvatar：创建灵眸 SDK 实例

`setupAvatar()` 是前端数字人初始化主流程：

```javascript
async function setupAvatar() {
  if (avatarInstance) return
  const sid = senderId.value.trim()
  if (!sid) return

  avatarState.value = 'connecting'
  let data = await fetchAvatarSession(sid)
  avatarSessionData = data
  avatarReady = false

  const avatar = createAvatar(TYAvatarType.cloudAvatar, {
    rootContainer: '#cloudAvatarContainer',
    ...data.rtcParams,
    sessionId: data.sessionId,
    ignoreAudioInput: true,
  })
  avatarInstance = avatar

  avatar.start({
    mode: TYVoiceChatMode.tap2talk,
    avatarAudioFormat: IAvatarAudioFormat.s16,
    outboundSampleRate: 16000,
  })

  avatar.onFirstFrameReceived(() => {
    avatarState.value = 'ready'
  })
  avatar.onReadyToSpeech(() => {
    avatarReady = true
  })
  avatar.onStateChanged((state) => {
    avatarState_TY = state
    flushTranscriptQueue()
  })

  openChatSocket()
}
```

按顺序看，它做了六件事：

| 顺序 | 动作 | 说明 |
| --- | --- | --- |
| 1 | `fetchAvatarSession(sid)` | 请求后端 `GET /api/avatar/session` |
| 2 | `avatarSessionData = data` | 保存后端返回的数字人会话参数 |
| 3 | `createAvatar(...)` | 创建云渲染数字人实例 |
| 4 | `avatar.start(...)` | 建立 SDK / RTC 通道 |
| 5 | 注册 SDK 事件 | 记录画面 ready、语音 ready、状态变化和错误 |
| 6 | `openChatSocket()` | 数字人初始化后，再打开后端聊天 WebSocket |

这里的顺序很重要：先有灵眸会话和 SDK 实例，再建立 `/ws/avatar/chat` 文本通道。否则前端即使收到了 `bot_text`，也还没有地方播报。

数字人挂载的 DOM 是页面里的：

```html
<div id="cloudAvatarContainer" class="avatar-video"></div>
```

`rootContainer: '#cloudAvatarContainer'` 会让 SDK 把云端渲染出来的数字人画面挂到这个区域。

### 9.4 打开聊天 WebSocket

数字人 SDK 启动后，前端调用 `openChatSocket()`：

```javascript
function openChatSocket() {
  if (chatSocket && (chatSocket.readyState === WebSocket.OPEN || chatSocket.readyState === WebSocket.CONNECTING)) {
    return
  }
  const proto = window.location.protocol === 'https:' ? 'wss' : 'ws'
  const url = `${proto}://${window.location.host}/ws/avatar/chat`
  wsState.value = 'connecting'

  const socket = new WebSocket(url)
  socket.binaryType = 'arraybuffer'
  chatSocket = socket

  socket.addEventListener('open', () => {
    wsState.value = 'open'
  })
  socket.addEventListener('message', (event) => {
    if (typeof event.data === 'string') {
      handleSocketJson(event.data)
    } else {
      handleSocketBinary(event.data)
    }
  })
  socket.addEventListener('close', () => {
    wsState.value = 'closed'
    rejectPendingTurn(new Error('connection closed'))
    chatSocket = null
  })
}
```

`socket.binaryType = 'arraybuffer'` 是为了兼容音频驱动模式留下的能力。当前 transcript 模式下，后端实际只发 JSON 文本，所以主要进入 `handleSocketJson()`。

### 9.5 发送消息：什么时候走 WS，什么时候走 HTTP

用户点击发送时，先执行 `sendTextMessage()`：

```javascript
async function sendTextMessage() {
  const text = draftMessage.value.trim()
  const currentSenderId = senderId.value.trim()

  if (!currentSenderId) {
    errorMessage.value = '请先输入 sender_id。'
    return
  }
  if (!text) return

  draftMessage.value = ''
  appendUserText(text)
  await sendPayload({ text })
}
```

这里先 `appendUserText(text)`，所以用户自己的气泡是前端立即显示的，不等后端返回。

然后进入 `sendPayload()` 分流：

```javascript
async function sendPayload(payload) {
  if (isSending.value) return

  errorMessage.value = ''
  isSending.value = true

  try {
    if (payload && payload.text && !payload.object) {
      if (!chatSocket || chatSocket.readyState !== WebSocket.OPEN) {
        await sendPayloadHttp(payload)
      } else {
        const messageId = crypto.randomUUID()
        await sendOverSocket(payload.text, messageId)
      }
    } else {
      await sendPayloadHttp(payload)
    }
  } finally {
    isSending.value = false
  }
}
```

分流规则是：

| 消息类型 | WebSocket 状态 | 实际调用 |
| --- | --- | --- |
| 纯文本 | 已连接 | `sendOverSocket()` -> `/ws/avatar/chat` |
| 纯文本 | 未连接 | `sendPayloadHttp()` -> `POST /api/chat` |
| 订单 / 商品对象 | 不管 WS 是否连接 | `sendPayloadHttp()` -> `POST /api/chat` |

所以 `POST /api/chat` 没有被数字人链路替代，它仍然承担两个职责：

1. 数字人 WebSocket 未连接时，保证纯文字聊天还能用。
2. 订单 / 商品对象消息仍然走原来的 HTTP 对话接口。

### 9.6 sendOverSocket：发送 user_text 前先打断旧播报

纯文本且 WebSocket 在线时，会进入：

```javascript
function sendOverSocket(text, messageId) {
  return new Promise((resolve, reject) => {
    if (!chatSocket || chatSocket.readyState !== WebSocket.OPEN) {
      reject(new Error('数字人通道未连接'))
      return
    }

    pcmPlayer.stop()
    clearPendingPush()
    clearTranscriptQueue()
    if (avatarInstance) {
      try { avatarInstance.interrupt() } catch (_) { /* ignore */ }
    }
    if (pendingTurn) {
      try { pendingTurn.reject(new Error('superseded')) } catch (_) { /* ignore */ }
      pendingTurn = null
    }

    pendingTurn = { messageId, resolve, reject, expectingAudio: false }
    chatSocket.send(
      JSON.stringify({
        type: 'user_text',
        sender_id: senderId.value.trim(),
        text,
        message_id: messageId,
      }),
    )
  })
}
```

这段代码和后端第 8 章的 `cancel_current()` 是配套的：

| 前端发送前 | 后端收到后 |
| --- | --- |
| 停止本地旧播放 | 取消旧 `current_turn` |
| 清空 transcript 队列 | 发送 `interrupt` |
| 调 `avatar.interrupt()` | 创建新的 `_run_turn` |
| 发送新的 `user_text` | 推送新的 `bot_text` |

也就是说，打断不是只在前端做，也不是只在后端做，而是两边一起保证“新问题优先”。

### 9.7 收到 bot_text：先显示，再排队播报

后端推回来的 JSON 都会进入 `handleSocketJson()`：

```javascript
function handleSocketJson(raw) {
  let payload
  try {
    payload = JSON.parse(raw)
  } catch (e) {
    console.warn('ws JSON parse failed:', raw)
    return
  }

  switch (payload.type) {
    case 'user_ack':
      break
    case 'bot_text':
      appendBotMessages([
        {
          text: payload.text || '',
          object: payload.object || null,
        },
      ])
      if (USE_TRANSCRIPT_MODE && payload.text) {
        transcriptQueue.push(payload.text)
        flushTranscriptQueue()
      }
      break
    case 'turn_end':
      if (pendingTurn) {
        const turn = pendingTurn
        pendingTurn = null
        turn.resolve()
      }
      break
    case 'interrupt':
      pcmPlayer.stop()
      clearPendingPush()
      clearTranscriptQueue()
      if (avatarInstance) {
        try { avatarInstance.interrupt() } catch (_) { /* ignore */ }
      }
      break
    case 'error':
      errorMessage.value = payload.message || '数字人对话出错'
      rejectPendingTurn(new Error(payload.message || 'server error'))
      break
  }
}
```

最重要的是 `bot_text` 分支，它做两件事：

```javascript
appendBotMessages([{ text: payload.text || '', object: payload.object || null }])
transcriptQueue.push(payload.text)
flushTranscriptQueue()
```

| 动作 | 结果 |
| --- | --- |
| `appendBotMessages` | 聊天窗口出现机器人气泡 |
| `transcriptQueue.push` | 同一段文本进入数字人待播队列 |
| `flushTranscriptQueue` | 如果 SDK 已空闲，就立刻让数字人播报 |

这就是“文字客服”和“数字人客服”的衔接点：同一个 `bot_text`，既用于页面展示，也用于驱动数字人开口。

### 9.8 transcript 队列：控制数字人播报节奏

灵眸 SDK 不是任何状态都能接受 `requestToRespond()`。如果数字人正在 `Thinking` 或 `Responding`，继续塞下一段文本可能失败。

所以前端用队列和状态门控：

```javascript
function flushTranscriptQueue() {
  if (!USE_TRANSCRIPT_MODE) return
  if (!avatarInstance || !avatarReady) return
  if (transcriptInFlight) return
  if (avatarState_TY !== 'Idle' && avatarState_TY !== 'Listening') return
  if (transcriptQueue.length === 0) return

  const text = transcriptQueue[0]
  transcriptInFlight = true
  transcriptSawActiveState = false
  try {
    avatarInstance.requestToRespond(IRequestToRespondType.transcript, text)
    transcriptQueue.shift()
  } catch (e) {
    transcriptInFlight = false
    transcriptSawActiveState = false
    console.warn('[avatar] requestToRespond failed:', e)
  }
}
```

它的门槛是：

| 判断 | 含义 |
| --- | --- |
| `USE_TRANSCRIPT_MODE` | 当前必须是文本驱动模式 |
| `avatarInstance` | SDK 实例已经创建 |
| `avatarReady` | SDK data channel 已经 ready |
| `transcriptInFlight` | 上一段 transcript 没播完时不塞下一段 |
| `Idle` / `Listening` | 数字人空闲或可接收输入 |
| `transcriptQueue.length` | 队列里确实有待播报文本 |

调用成功后：

```javascript
avatarInstance.requestToRespond(IRequestToRespondType.transcript, text)
```

这一步不是发给 FastAPI，而是通过灵眸 SDK 发给阿里云。后续链路是：

```text
阿里云收到 transcript 文本
    -> 服务端 TTS 合成声音
    -> 生成唇形和表情
    -> 云端渲染数字人画面
    -> 通过 RTC 音视频流回到浏览器
    -> 用户看到数字人开口
```

当 SDK 状态变化时，前端会继续推进队列：

```javascript
avatar.onStateChanged((state) => {
  avatarState_TY = state
  if (state === 'Thinking' || state === 'Responding') {
    transcriptSawActiveState = true
  }
  if ((state === 'Idle' || state === 'Listening') && transcriptInFlight && transcriptSawActiveState) {
    transcriptInFlight = false
    transcriptSawActiveState = false
  }
  flushTranscriptQueue()
})
```

这样可以保证多条 `BotMessage` 按顺序播报，而不是一起塞给 SDK。

### 9.9 数字人会话释放

数字人会话释放只针对阿里云灵眸云渲染会话本身，不和业务用户 `sender_id` 绑定。

当前设计里，一个数字人实例在后端进程内只维护一个共享 session。不同用户的对话身份仍然由 WebSocket 消息里的 `sender_id` 区分；数字人 session 只负责 RTC 画面、声音和唇形渲染。

释放接口如下：

| 方法 | 路径 | 参数 | 用途 |
| --- | --- | --- | --- |
| DELETE | `/api/avatar/session` | 无 | 释放当前进程缓存的数字人 session，归还阿里云并发位 |

释放时后端会读取当前缓存的 `sessionId`，调用灵眸 `CloseChatInstanceSessions` 关闭会话。释放成功后，下一次访问 `GET /api/avatar/session` 会重新创建一个新的数字人 session。

这里不再使用 `DELETE /api/avatar/session?sender_id=...`。`sender_id` 是业务对话身份，不是数字人云渲染会话的释放依据。

## 第 10 章 历史对话展示

前面几节已经把电商客服的核心对话链路跑通了：用户发消息，后端进入 `DialogueEngine`，再根据 planner 的结果走 task、knowledge、chitchat 等不同轨道，最后把机器人回复返回给前端。

但是，一个真正可用的客服系统不能只处理"当前这一句话"。用户刷新页面、重新进入页面时，也应该看到之前的聊天记录。

这一节要补上的就是**历史对话展示能力**。

### 10.1 任务目标

目标效果分成两种场景：

1. **用户刚进入页面加载历史**
   - 返回数据库里所有 session 的消息
   - 前端直接展示全部历史
   - 不插入分割线

2. **用户进入页面后发送了新消息**
   - 页面加载时已有的消息算历史
   - 本次页面打开后新产生的消息算当前消息
   - 这时才在两者中间插入分割线

也就是说，分割线是前端根据"页面生命周期"做出来的展示效果。

### 10.2 设计路线

历史对话这块最容易混淆的点是：**session 是后端业务概念，分割线是前端展示概念**。

后端知道每条消息属于哪个 session，但后端并不知道用户是不是"刚刚打开页面"。页面打开前后的边界，是浏览器里的状态，不是数据库里的状态。

所以这一节采用的设计是：

| 层 | 职责 |
| --- | --- |
| 后端 | 返回所有历史消息，并给每条消息带上 `session_id` |
| 前端 | 记录页面首次加载时的历史快照，首次发送新消息时插入分割线 `divider` |

这条分工非常重要：

- `session_id` 用来表达"消息属于哪个后端会话"
- `divider` 用来表达"页面打开前"和"页面打开后"的视觉边界

两者不是同一个概念。

那为什么后端还要返回 `session_id` 呢？因为历史消息接口返回的是一组平铺消息，如果不带 `session_id`，前端虽然能展示消息，但已经看不出这些消息原本属于哪个后端 session。

例如后端原始结构是：

```text
session-a
  用户：查订单
  机器人：请提供订单号

session-b
  用户：我要退款
  机器人：请问退款原因
```

如果接口只返回 `role` 和 `text`，这个 session 归属关系就丢了。加上 `session_id` 后，即使消息被平铺返回，也还能保留"每条消息属于哪个业务会话"的信息。

所以可以这样理解：

| 概念         | 作用                                         |
| ------------ | -------------------------------------------- |
| `session_id` | 后端数据归属，表示消息属于哪个业务 session   |
| `divider`    | 前端展示效果，表示页面加载前后消息的视觉分割 |

对当前"首次进入不显示分割线，首次发送才显示分割线"的功能来说，`session_id` 不是必需条件；但对历史接口来说，返回 `session_id` 是合理的，因为它保留了后端会话结构，方便后续做按会话折叠、会话分组展示等扩展。

有了这个分工，下面先从后端的数据模型开始讲。

### 10.3 后端响应模型

历史接口返回的消息，和普通 `/api/chat` 返回的机器人回复不完全一样。

普通聊天接口只需要返回机器人本轮回复：

```python
class ChatResponse(BaseModel):
    sender_id: str
    message_id: str
    messages: list[ChatBotMessage]
```

历史接口则要返回很多轮消息，而且每条消息既可能是用户说的，也可能是机器人说的，所以单独定义一个 `ChatHistoryMessage`。

```python
@dataclass(slots=True)
class ChatHistoryMessage:
    session_id: str
    role: Literal["user", "bot"]
    text: str | None = None
    object: FocusedObject | None = None
```

这个模型里最关键的是两个字段：

| 字段 | 说明 |
| --- | --- |
| `session_id` | 这条历史消息属于哪个后端 session |
| `role` | 消息角色，只允许 `user` 或 `bot` |

历史接口的整体响应模型如下：

```python
class ChatMessageResponse(BaseModel):
    sender_id: str
    messages: list[ChatHistoryMessage]
```

现在模型定好了，下一步就是把数据库中的 session / turn 转成这个响应模型。

### 10.4 构建历史消息

项目里原来已经有一个 `HistoryBuilder`，它主要负责把历史对话渲染成 prompt 字符串，供 LLM 理解上下文。

比如：

```python
USER: 我想查订单
BOT: 请提供订单号
```

这次历史接口要做的事情不一样。它不是给 LLM 看，而是给前端页面看，所以我们在 `HistoryBuilder` 里增加两个面向接口响应的方法。

```python
@staticmethod
def render_user_message(session_id: str, user_message: UserMessage) -> ChatHistoryMessage:
    return ChatHistoryMessage(
        session_id=session_id,
        role="user",
        text=user_message.text,
        object=user_message.object,
    )

@staticmethod
def render_bot_messages(session_id: str, bot_messages: BotMessage) -> ChatHistoryMessage:
    return ChatHistoryMessage(
        session_id=session_id,
        role="bot",
        text=bot_messages.text,
        object=bot_messages.object,
    )
```

这两个方法做的事情很轻：

| 方法 | 输入 | 输出 |
| --- | --- | --- |
| `render_user_message` | `session_id` + `UserMessage` | role 为 `user` 的历史消息 |
| `render_bot_messages` | `session_id` + `BotMessage` | role 为 `bot` 的历史消息 |

这里把 `session_id` 作为参数传进去，而不是让 `UserMessage` 或 `BotMessage` 自己保存 session 信息。

原因是：

- `UserMessage` 表示"用户发来的这一条消息"
- `BotMessage` 表示"机器人回复的一条消息"
- 它们本身不负责知道自己属于哪个 session
- session 归属关系存在于外层 `Session.turns` 结构里

所以转换历史消息时，由外层遍历 session 的代码把 `session.session_id` 补进去。

有了单条消息的转换方法，下面进入 service 层，把所有 session 展开。

### 10.5 加载全部历史

历史加载入口在 `DialogueService.load_chat_history`。

```python
async def load_chat_history(self, sender_id: str) -> list[ChatHistoryMessage]:
    state = await self.dialogue_repository.load(sender_id)

    chat_messages: list[ChatHistoryMessage] = []
    self._append_session_history_message(state.sessions, chat_messages)

    return chat_messages
```

这段代码只有三步：

| 步骤 | 做什么 |
| --- | --- |
| 1. load state | 根据 `sender_id` 从数据库加载完整 `DialogueState` |
| 2. 展开 sessions | 遍历 `state.sessions`，把每个 turn 展开成消息列表 |
| 3. 返回列表 | 返回给 router 组装接口响应 |

真正展开消息的是下面这个方法：

```python
def _append_session_history_message(self, sessions: list[Session],
                                    chat_messages: list[ChatHistoryMessage]):
    for session in sessions:
        for turn in session.turns:
            chat_messages.append(HistoryBuilder.render_user_message(session.session_id, turn.user_message))
            chat_messages.extend([
                HistoryBuilder.render_bot_messages(session.session_id, b_msg)
                for b_msg in turn.bot_messages
            ])
```

它的遍历顺序是：

```text
Session 1
  Turn 1
    user message
    bot message 1
    bot message 2
  Turn 2
    user message
    bot message
Session 2
  Turn 1
    user message
    bot message
```

展开后，接口返回的是一个平铺的消息数组。

示例响应大概长这样：

```json
{
  "sender_id": "u1001",
  "messages": [
    {
      "session_id": "session-a",
      "role": "user",
      "text": "我想查订单",
      "object": null
    },
    {
      "session_id": "session-a",
      "role": "bot",
      "text": "请提供订单号。",
      "object": null
    },
    {
      "session_id": "session-b",
      "role": "user",
      "text": "我要退款",
      "object": null
    }
  ]
}
```

历史接口定义在 `chat_router.py`。

```python
@router.get("/api/chat/history", response_model=ChatMessageResponse)
async def chat_history_endpoint(sender_id: str,
                                service: DialogueService = Depends(get_dialogue_service)) -> ChatMessageResponse:
    chat_message_response: list[ChatHistoryMessage] = await service.load_chat_history(sender_id)

    return ChatMessageResponse(sender_id=sender_id, messages=chat_message_response)
```

### 10.6 前端状态设计

前端核心状态在 `App.vue` 里。

```js
const messages = ref([])
const messagesContainer = ref(null)
const loadedHistoryCount = ref(0)
const currentPageDividerInserted = ref(false)
```

这几个字段分工如下：

| 状态 | 说明 |
| --- | --- |
| `messages` | 页面当前要渲染的完整消息列表 |
| `loadedHistoryCount` | 页面首次加载历史后，历史消息的数量 |
| `currentPageDividerInserted` | 本次页面是否已经插入过分割线 |

重点是后两个字段。

`loadedHistoryCount` 表示"页面打开时已经存在多少条历史消息"。如果它是 0，说明这个用户没有历史消息，那么用户第一次发送消息时也不需要插入分割线。

`currentPageDividerInserted` 用来保证分割线只插入一次。否则用户每发一条消息都插一条，就变成噪音了。

有了这两个状态，前端就能准确表达业务边界：

```text
页面加载前已有消息：历史
页面加载后追加消息：当前
```

### 10.7 完整交互流程

#### 10.7.1 用户刚进入页面

```text
前端 onMounted
    ↓
fetchChatHistory()
    ↓
GET /api/chat/history?sender_id=u1001
    ↓
后端加载 state.sessions
    ↓
后端返回所有历史消息，每条带 session_id
    ↓
前端 setHistoryMessages
    ↓
messages = 全部历史消息
loadedHistoryCount = 历史消息数量
currentPageDividerInserted = false
    ↓
页面直接展示全部历史，不显示分割线
```

这个阶段的重点是：**不插入分割线**。

即使历史消息来自多个 session，也全部作为历史展示。

#### 10.7.2 用户发送本页第一条新消息

```text
用户输入：我要退款
    ↓
sendTextMessage()
    ↓
appendUserText(text)
    ↓
insertCurrentPageDividerIfNeeded()
    ↓
发现 loadedHistoryCount > 0 且还没插入过 divider
    ↓
插入 "以上为历史消息"
    ↓
追加用户新消息
    ↓
调用 /api/chat 或 WebSocket
    ↓
机器人回复 appendBotMessages
```

这时页面结构变成：

```text
历史消息 1
历史消息 2
历史消息 3
------ 以上为历史消息 ------
本页用户新消息
本页机器人回复
```

#### 10.7.3 用户继续发送后续消息

```text
用户再次发送消息
    ↓
insertCurrentPageDividerIfNeeded()
    ↓
currentPageDividerInserted 已经是 true
    ↓
不再插入 divider
    ↓
直接追加新消息
```

页面结构保持清爽：

```text
历史消息
------ 以上为历史消息 ------
本页第一轮
本页第二轮
本页第三轮
```

### 11.1 这一节实现了什么

这一节做了两件事：把原来的文字客服升级成了数字人客服，同时补齐了历史对话展示能力。

### 11.2 几个关键设计

**数字人部分：**
1. **后端复用原有对话引擎**：数字人没有新写一套业务逻辑，只是把 `BotMessage.text` 通过 WebSocket 多送给前端。
2. **WebSocket 只传文本**：当前使用 transcript 模式，TTS 和唇形由灵眸服务端完成，后端不合成 PCM。
3. **HTTP 会话和 WS 对话分离**：HTTP 负责创建 SDK 会话，WS 负责对话过程，RTC 负责音视频。
4. **前端用 transcript 队列控节奏**：只有数字人处于 `Idle` 或 `Listening` 时才调用 `requestToRespond`。

**历史对话部分：**
5. **session_id 与 divider 分离**：`session_id` 是后端数据归属概念，`divider` 是前端展示概念。后端不知道页面何时打开，所以分割线由前端根据页面生命周期插入。
6. **分割线只插入一次**：通过 `loadedHistoryCount` 和 `currentPageDividerInserted` 两个状态保证首次发送新消息时插入，后续不再重复。
7. **历史接口复用 DialogueState**：不新建存储结构，直接从现有 `state.sessions` 展开返回，保持数据源单一。

到这里，电商小二已经不只是能”回答”，还可以通过阿里云数字人”说出来”，并且用户刷新页面后能看到完整的聊天记录。
